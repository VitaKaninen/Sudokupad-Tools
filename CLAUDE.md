TamperMonkey userscript that fixes DarkReader / dark-theme visual issues on SudokuPad (`sudokupad.app` and related domains) and adds quality-of-life features. Single file: `sudokupad-native-dark-mode.user.js`. Tested on Chrome + TamperMonkey + DarkReader.

## Standing instructions

### Cross-referencing the puzzle catalog (side-effect check)
`docs/Catalog/` inventories 1890 puzzles and the render buckets they contain, to predict whether a **broad change breaks other puzzles**. It is a tool to reach for *deliberately*, not by default — querying it sloppily or when it can't help wastes tokens (its whole point is to *save* the user from blind broad changes, so don't undermine that).

- **When to use it:** before a change that touches a **shared bucket** many puzzles have (`#arrows | path`, overlay rects, `#cell-colors`, kropki/`textbg`, cages) — i.e. not scoped to one puzzle. Query it to count affected puzzles, check the attribute-union for a variant that breaks your assumption, and pull 3–5 URLs to spot-check. **Don't** consult it for a single-puzzle tweak, a CSS/UI-only change, or anything that can't reach other puzzles.
- **How to use it without burning tokens — never read the files into context** (raw is 8 MB; union ~80k tokens). Always query via `python` and return only the small answer. **For the CSV use Python's `csv` module, never `awk -F,`/`grep`-by-column** — quoted commas in titles silently mis-align columns and give wrong counts. Exact query patterns + caveats: see **"Puzzle catalog"** in [`docs/PROJECT_SUMMARY.md`](docs/PROJECT_SUMMARY.md).

### Commit + push after every edit
After every version bump, immediately and without asking:
1. `git add sudokupad-native-dark-mode.user.js` (and any docs you changed)
2. `git commit -m "vX.Y.Z: <short description>"`
3. `git push`

### Version bumps
Bump `@version` in the `==UserScript==` header for every change — semver minor increments (e.g. 2.119.0 → 2.120.0). **Also bump the internal `SCRIPT_VERSION` constant (≈line 209) to match** — it drives `window.spdrVersion`, the only reliable "which build is live" signal in-browser. It silently drifted (stuck at 3.11.0 while the header advanced); keep them identical.

### Testing in Chrome (standing permission)

**Diagnosing up front is yours; the first "does it work now" test is the user's.** Connect Claude in Chrome any time it helps you *diagnose* a problem — don't ask. Say upfront when the browser would answer better than guessing (e.g. pixel-level rendering). If you need a screenshot to understand an issue, take it yourself: the cost is in *analysing* the image, not capturing it, so having the user paste one in saves nothing.

But **do not verify your own fix in-browser.** Self-testing a change burns a lot of reload/inspect/screenshot round-trips, and that's expensive. Once you've made and committed a change, let the user run the initial check and report back. **Don't narrate "what to expect" by default** — commit and give a brief "done". Only describe the expected visible result when the effect is genuinely non-obvious or easy to misread; for straightforward changes, skip it. Only go back to the browser yourself if the user reports it's still wrong and you need to inspect (or they ask you to). At that point more screenshots/tests are fine.

**Your working tab is yours alone — the user will not touch it.** The user deliberately does *not* refresh or change the tab you're driving, so you never lose track of its state; take full control of it and they'll keep their own changes to a minimum. **Cross-session caveat (learned 2026-06-06):** the Claude-in-Chrome MCP tab group is *per-conversation* — a NEW session cannot adopt the tab a previous session drove (`tabs_context_mcp` reports "no tab group exists for this session" even after `select_browser`). So if the user says "reuse the tab you already have open," that's only possible *within* the same session; across sessions you must `tabs_context_mcp({createIfEmpty:true})`, which spawns a fresh **isolated new window** (safe from the user's other windows — it can't touch them) — then `tabs_close_mcp` it when done so nothing lingers. Don't burn turns hunting for the old tab; explain briefly and create the fresh one. (This is the opposite of the user's *own* testing tabs, which they reload freely.) Reload your tab yourself whenever you need the latest script live — TamperMonkey serves the file on its own poll, so a just-saved edit may not be running yet; confirm via `window.spdrVersion`. Never ask the user "did you reload?": when the user reports an issue, they have **already** reloaded and confirmed the correct script version is running before telling you.

**Flipping native↔main is self-serve via the A/B harness; only the DarkReader *extension* isn't.** Both scripts stay enabled in TamperMonkey, gated per-tab by `#variant=a|b` (the A/B comparison rig — see [PROJECT_SUMMARY](docs/PROJECT_SUMMARY.md) → Testing setup). To switch the live script, navigate your own tab to `…#variant=a` (DarkReader Fix 2.x) or `#variant=b` (Native 3.x) and reload — **don't ask, just do it.** ⚠️ **Gotcha: a hash-only navigation does NOT reload the page** (the URL path is unchanged, so the browser just updates the fragment and the userscript keeps running the OLD variant). After setting `#variant=…` you MUST explicitly force a reload — `location.reload()` via the JS tool, or navigate to the bare URL again — *then* re-check `window.spdrVersion`/`spdrEdition` to confirm the new variant actually took. Don't query state right after the hash nav and trust it. For a plain visual comparison still prefer asking (via popup) for a screenshot of the state you need rather than driving the tab; drive the flip yourself when you need *computed values* (exact `rgb()`, border widths) a screenshot can't give. The one state you can't set yourself is the **DarkReader browser extension** on/off — if a task genuinely needs DR toggled, ask the user. (`window.spdrEdition` is `'native'` vs absent; `window.spdrVersion` tells 2.x vs 3.x.)

**Stale TamperMonkey version → stop and ask, don't give up.** If you reload your tab and `window.spdrVersion` is still *older than the repo's `@version`* (TM served a stale script), do **not** throw up your hands, work around it, or silently proceed against the old version. The cause is almost always the user's change-tracking tab going to sleep and just needing a close-and-reopen — a minor fix on their end. Surface it via `AskUserQuestion` so they get the chance to fix it: a single option **"Try again" (the only real choice — plus the write-in)**, e.g. question "TamperMonkey is still serving stale vX.Y.Z (repo is vA.B.C). I'll wait while you reload the tracking tab." The user will fix it, then answer "Try again" — that answer means *they expect it to work now*, so reload the tab and re-check the version. If it's **still** stale after that reload, ask the **same question again** (they need another opportunity to fix it). Repeat until the version matches. Only deviate from "Try again" if they write in something else.
