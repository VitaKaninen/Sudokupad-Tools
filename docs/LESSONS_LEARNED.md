*Flat list of what's confirmed to work, what's confirmed NOT to work, and why. The point of this file: never re-fight a solved battle. When a session confirms a new fix or a dead end, add a line. Record the conclusion, not the journey.*

## Native dark mode ‚Äî the substrate swap (v2.195, in progress on branch `native-mode`)

**Direction change:** stop fighting DarkReader; evict it from SudokuPad and ride the site's own native dark mode instead. DR is a blind, adversarial colour *inverter* that re-runs on every mutation; native dark mode is the author's *semantic*, *static* stylesheet. Below "Beating DarkReader" describes the OLD architecture (the `main` branch). This section is the new one.

- ‚úÖ **`<meta name="darkreader-lock">` evicts DR cleanly ‚Äî confirmed in the user's DR build.** Injected *dynamically* at runtime, DR tore itself down within ~1.2 s: `data-darkreader-scheme` attr removed, **all** `style.darkreader` tags removed (43 ‚Üí 0), `--darkreader-*` vars gone. The leftover `data-darkreader-inline-fill` *attributes* on elements become inert orphans (no rule references them). Our top-of-IIFE `lockDRUseNative()` block injects the meta at `document-start` (head may not exist yet ‚Üí MutationObserver fallback). DR keeps working on every other site. (NB a DR regression broke the lock in build v4.9.86 ‚Äî verify per-environment; the user's build works.)
- ‚úÖ **Native dark mode = the `darkmode` flag inside the `svencodes_settings` localStorage object ‚Üí `body.setting-darkmode` class.** Our block sets the flag true (so SudokuPad's own toggle reflects it + re-applies on init) AND adds the class live (no reload / load-order dependence). **Side effect: this persistently flips the user's SudokuPad `darkmode` setting on** (reversible in SudokuPad settings).
- **Native palette** (`.setting-darkmode` CSS vars): `--dm-black:#1a1a1a`, `--dm-white:#eee`, `--dm-button-color:#8522c3` (purple), `--dm-rulesbg:#265016`, `--dm-userblue:#5f95ec`, `--dm-black-alpha:rgba(18,18,18,.7)`.
- **What native dark mode HANDLES (for free, correctly):** `.cell-grid`/`.cage-box` strokes ‚Üí `--dm-white`; white labels via `rect.textbg[fill="#FFFFFF"]` (+ case variants) & `.textbg_ffffff` ‚Üí `--dm-black`; `[bordercolor="#fff"‚Ä¶]`; givens (`.cell-given`/`.givenCornermark`/`.cell-candidate .given`) ‚Üí `--dm-white`; `.dialog`, `#controls` text, `.puzzle-rules`, settings toggles/labels, `#fog-fogcover`.
- ‚öÝÔ∏è **What native dark mode MISSES ‚Üí the 3 known gaps the rest of the script must fix:** (1) **arbitrary gray / translucent-dark objects aren't inverted** ‚Äî cell-shading authored as translucent black (`#0003`) and dark grays stay dark ‚Üí vanish on the dark page (DR *did* invert these). ‚Üí our **Object Shading** covers it, and is *more* needed here, not less. (2) **white labels that aren't an exact `[fill="#FFFFFF"]` match** ‚Äî translucent-white `#ffffff80` endpoints, class-less white circles ‚Üí stay white, white text unreadable. ‚Üí our **label-bg fix** is a broader white/near-white match (currently its native CSS branch keys on `body.setting-uitheme-purple`; re-key to `setting-darkmode`). (3) **control buttons stay bright** ‚Äî native themes `#controls` *text* only; `#control-settings`/`-fullscreen`/`-rules` keep `background:#eee`. ‚Üí needs one small new darkening rule.
- **Boot gate moved off DR:** the script used to gate its whole startup on `isDarkReader()` (`data-darkreader-scheme="dark"`). Now `isDarkMode()` = `isNativeDark()` (`body.setting-darkmode`) ‚Äñ `isDarkReader()`, so it boots on native and still works if DR somehow remains. `waitForDRAndSVG` also observes the body `class` attr (native lands there).
- **Migration shape (still TODO):** re-base DR-sampled defaults (`#181A1B`/`#dddad6`/`#e8e6e3`) ‚Üí native palette; re-key the label-bg CSS to `setting-darkmode`; generalize/remove the `inDR` branches in `fixCageBox` etc.; **amputate the entire `captureDrInline`/`restoreToDr`/`spdrDrFill` restore-to-DR machinery** (no DR var to restore to ‚Äî on disable just clear our inline & let native CSS show); handle `--cell-color-*` swatches under native (the `[data-darkreader-scheme]` specificity-bump rule is DR-only).

## Beating DarkReader (the core problem)

DarkReader overrides SVG fills two ways:
1. **Generated CSS** ‚Äî counter-rules in `<style class="darkreader darkreader--override">`, often the *same* specificity as ours but *later* in the document, so they win on cascade order.
2. **Inline attribute override** ‚Äî adds `data-darkreader-inline-fill` plus an `!important` stylesheet rule driven by `--darkreader-inline-fill`.

- ‚úÖ **The master key ‚Äî inline style with `!important`:** `el.style.setProperty('fill', value, 'important')`. DarkReader does **not** fight `!important` inline styles. Used for given digits, Kropki dots, underlay fills, pencilmarks, cell colours.
- ‚úÖ **CSS variables ‚Äî restore via `html[data-darkreader-scheme="dark"]`** (specificity 0,1,1), which beats DarkReader's `:root` block (0,1,0) for `--cell-color-*`.
- ‚ùå **CSS `!important` alone does NOT work:** DarkReader generates an equal-specificity counter-rule that appears *after* ours and wins. Always use inline-style `!important` for fills, never a stylesheet rule.
- ‚ùå **Don't strip `data-darkreader-inline-fill` from all rects:** it breaks the textbg-rect fix. No blanket removal.
- ‚úÖ **Turning a shading feature OFF must repaint DR's converted colour as a RESOLVED LITERAL** (v2.145 ‚Üí corrected v2.149). Once we set an `!important` inline colour, DR gives up on the element **and clears its own `--darkreader-inline-*` var** (verified) ‚Äî so `removeProperty('fill')` on disable leaves the raw, bright attribute colour, not DR's dim value. We must repaint DR's dark colour ourselves: `captureDrInline` stores it on `dataset.spdrDrFill`/`spdrDrStroke`, `restoreToDr` paints it back with `!important` on disable (it never touches the data-attribute, so the `data-darkreader-*` observer doesn't loop).
  - ‚ùå **The v2.149 bug fix corrects a v2.145 mistake: do NOT store/restore DR's var EXPRESSION** (`var(--darkreader-background-cfcfcf,#333739)`). Setting that as `style.fill` makes DR's observer see the `#333739` hex and re-convert it into a *text*-colour var (which for a dark input resolves *light*), nesting one level deeper and drifting **brighter** every toggle/redraw. That was the "disable makes it brighter / enabling region borders brightens the shaded areas" regression.
  - ‚úÖ **Store a resolved literal rgb instead.** Resolve a colour expression via a throwaway `<span>` (`style.color = expr` ‚Üí `getComputedStyle().color`); DR's vars live on `:root` so `var()` resolves synchronously through the cascade. Capture **once** (first clean value wins; the original colour never changes) so a later DR-derived value can't overwrite it. Deterministic fallback when the element's own DR var isn't present yet (scan-vs-DR race): DR publishes `--darkreader-<kind>-<hex>` root vars keyed by the element's stable original colour (`background` for fills, `text` for strokes), so build that reference from the `fill`/`stroke` attribute and resolve it. (a) restoring only the DR var *without* its data-attribute renders nothing ‚Äî DR's rule is keyed on the attribute selector, which we removed. (b) our `!important` inline literal beats DR's stylesheet rule, so it wins whether or not DR's marker is still there.

## SVG rendering

- ‚úÖ **`shape-rendering="crispEdges"` on our group** (fixed v2.82): the SVG is scaled ~1.333√ó (810px screen √∑ 608px viewBox), so strip boundaries land on fractional screen pixels; the browser anti-aliases the edge into the dark cell behind it ‚Üí a thin dark fringe. crispEdges snaps boundaries to whole pixels and the fringe disappears.
- ‚öÝÔ∏è **crispEdges on mainGroup must NOT reach the cloned `path.cell-grid`** (fixed v2.146). The grid clone (z-order fix) is a child of mainGroup, so it inherited `shape-rendering:crispEdges`. crispEdges is right for the strip *rects* (kills the fringe) but wrong for the thin 1px grid lines: at ~1.33√ó scale each line is ~1.33px and crispEdges snaps it to 1px or 2px depending on sub-pixel position ‚Üí some dividers render brighter/wider than others, with the rounding differing per browser (LibreWolf vs Brave hit different lines), and worst over shaded underlay cells. The original `path.cell-grid` was `shape-rendering:auto`. Fix: set `shape-rendering:geometricPrecision` on the *clone* to override the inherited crispEdges back to smooth. Lesson: anything thin/anti-aliased that we move into the crispEdges group needs its shape-rendering reset.
- ‚úÖ **`stroke: none !important` on the 9 cage-box paths** (fixed v2.81) whenever any feature is active: those `#cell-grids path:not(.cell-grid)` paths have opaque-black stroke at z=8 and otherwise paint dark lines through coloured cells at 3√ó3 box boundaries. Restore the stroke when all features are off.
- ‚úÖ **Cell-grid z-order fix** (v2.85): clone `path.cell-grid` as the *first* child of our group (so strips cover the clone) **and clear the original's `d` attribute** so it stops rendering at z=8. Restore the original from `dataset.spdrOrigD` on the next call.
- ‚ùå **Never use `display:none` (or any style change) on `path.cell-grid`:** the `#cell-grids` MutationObserver watches `attributeFilter: ['style']`. A style change fires it ‚Üí infinite loop. Changing `d` does **not** trigger it ‚Äî that's exactly why we clear `d` instead.
- ‚úÖ **White-stroked `#cell-grids path:not(.cell-grid)` are author "grid-line erasers", not box outlines (v2.192).** Most puzzles' non-`.cell-grid` paths in `#cell-grids` are the dark `#000000` 3√ó3/region box outlines we suppress + redraw. But some authors lay a **`stroke="#FFFFFF"` width-3** path *over* a grid line to HIDE it ‚Äî invisible white-on-white in light mode, used to carve irregular outer shapes / merge cells (e.g. the Gattai puzzle `y22px1zdae`: a 10√ó10 grid whose play area is two overlapping sub-grids, the rest erased). `fixCageBox` assumed black outlines, so it broke both ways: with region borders on it set `stroke:none` ‚Üí **revealed** the grid line the author hid; with them off it restored DR's `--darkreader-text-000000` (#e8e6e3) ‚Üí painted a **bright** thick line. **Fix:** detect a near-white stroke attr (r,g,b ‚â• 240, a‚âÝ0) and, under DR, repaint it `var(--darkreader-background-ffffff, #181a1b)` at native width so it keeps **erasing** the (now light) grid line ‚Äî exactly what SudokuPad's own dark mode does via `.setting-darkmode [stroke="#FFFFFF"]{stroke:var(--dm-black)}`. Discriminator is stroke COLOUR (white = eraser, dark = real outline), which is safe: a *visible* border is never white (invisible on the light default), so white in `#cell-grids` always means eraser.
- ‚öÝÔ∏è **`y22px1zdae`'s thick "region borders" are COSMETIC lines, not region/box data ‚Äî deliberately left alone (v2.192).** The puzzle's visible region structure is drawn as 15 thick black cosmetic polylines in `#overlay` (`stroke=#000000`, `stroke-width=5.1`); there is **no** region/box structure in the model (the 20 `(none)` cages are just the row+col uniqueness constraints `r1c1-r1c10`/`r1c1-r10c1`, and the 4 dashed `#cages` paths are cosmetic overlap reminders). DarkReader already converts those black lines to light thick lines, so they render fine and read as borders against the 1px grid ‚Äî they're just thicker, same colour. Our region-split feature can't "recognize" them because inferring regions from arbitrary cosmetic `#overlay` lines has no reliable signal (cosmetic black lines are decoration on countless puzzles) ‚Üí high blast radius across the ~100 vetted puzzles. **Decision (confirmed with the user): leave them.** Anything we draw over them is more confusing than the already-fine DR rendering. So the region-split feature stays inert here by design; only the eraser fix above applies.
- ‚úÖ **Region 4-colouring must use backtracking, not plain greedy (v3.12.0).** `computeRegion4Colors` builds the orthogonal-edge adjacency graph (planar ‚Üí four-colour theorem guarantees a proper 4-colouring exists) and feeds BOTH features off one shared `regionColors[]` array (multi-colour borders use `palette[ci]`, Easy Shade fills use `fillPalette[ci]` ‚Äî same indices, so they always match; the Easy Shade button only flips `regionColorFillEnabled`, it computes nothing itself). The original code did **plain greedy in row-major order**, which is *proper* (never colours two touching regions alike) but does **not** stay within 4 colours on irregular/jigsaw region layouts ‚Äî greedy can spill to a 5th colour (index 4). The palette has only 4 entries, so `palette[4]` is `undefined` ‚Üí the region's strips/fills render with **no fill = black borders** (the user hit this on real puzzles). Fix: greedy fast-path, and if it overflows past 0‚Äì3, run a backtracking proper-4-colouring (guaranteed to succeed for a planar graph; region counts are tiny so it's instant), plus a clamp-to-0 final guard so an index can never exceed the palette. Standard 3√ó3-box sudoku never tripped it (that graph is 2-colourable); only irregular regions did.
- ‚úÖ **Disjoint/scattered sudoku regions need the puzzle MODEL, not geometry (v3.13.0).** Some puzzles (e.g. clover's "Scattered" `2xjf6aw4m2`) define one sudoku region whose 9 cells are **not contiguous** ‚Äî scattered single cells that are one region for digit logic and must share one colour. `inferRegionsFromSVG` is pure geometry (flood-fill between border paths) and CAN'T know non-touching pieces are one region ‚Üí it saw them as 9 separate regions in 9 colours. **The logical region list is in the runtime model:** `app.puzzle.currentPuzzle.cages`, filtered to `type === 'region'`, each with a comma-separated **1-indexed** RC-notation `cells` string (`"r2c2,r5c8,‚Ä¶"`). NB: `app.puzzle.cells[i]` does NOT carry a `region` field (only row/col/value/‚Ä¶); the resolved region membership lives only on the `currentPuzzle` cage list. Normal sudoku has **no** `type:'region'` cages (boxes are implicit) ‚Üí map is null ‚Üí geometry path unchanged. Implementation: `getModelRegionMap` caches `{ 'r,c': logicalId }` keyed by `location.pathname`; since the render path is sync but `Framework.getApp()` is async, first call returns null + kicks an async refresh that repaints once when the map lands (guarded by a `_pending` flag + key-match so no loop). `buildRegionGroups` collapses geometric pieces with the same logical id into one colour group; `computeRegion4Colors` now colours per GROUP and returns per geo-region (drawing code unchanged ‚Äî disjoint pieces just get matching colours and each still draws its own border outline). ‚öÝÔ∏è **Forcing a DISCONNECTED region to one colour can break 4-colourability** (the four-colour theorem only guarantees 4 for *connected* regions) ‚Äî so `computeRegion4Colors` falls back to ungrouped (geometry) colouring when no proper grouped 4-colouring exists, preferring a clash-free result over the disjoint match in that rare case. (Single-disjoint-region puzzles are usually still 4-colourable: the scattered region touches everything ‚Üí just takes the 4th colour.) **v3.14.0: disjoint groups are PINNED to palette colour 3 (the 4th, orange)** so the common regions keep the earlier, more-frequent colours (red 0 > blue 1 > green 2) and the scattered region is the rare orange. `countComponents` flags a group as disjoint when its cells form >1 orthogonal component; `colourGraph` takes an optional `pinned[]` (fixed colour per node, rejects mutually-clashing pins, backtracks only the free nodes). Degrade order: pinned grouped ‚Üí unpinned grouped ‚Üí ungrouped geometry. (Pinning the scattered node to 3 forces all its neighbours off 3, so the rest must be 3-colourable with {0,1,2} ‚Äî held for `2xjf6aw4m2`; if it ever doesn't, it falls to unpinned automatically.)
- ‚ùå **Dead end ‚Äî `fixCellGrid`** (the v2.71‚Äìv2.80 clipping function): tried to clip cell-grid segments at region boundaries. Removed v2.83. The "white lines in black cells" it chased were actually the cage-box strokes (fixed v2.81) + sub-pixel anti-aliasing (fixed v2.82). v2.67 had neither `fixCellGrid` nor the white-lines problem ‚Äî confirmed baseline.

## Region / shaded-region geometry

- ‚úÖ **`path.cage-extraregion` only ever means a *checked* region (digit-uniqueness enforced), never cosmetic shading.** Verified via `window.convertedPuzzle.cages`: those paths come from cages with `style:"extraregion", unique:true, sum:45` (a real 1-9 no-repeat constraint). The cage `style` tally on a normal puzzle is `region` (the 9 boxes), `rowcol` (rows+cols), and `extraregion`. Purely decorative grey shading lives in a **separate `cosmetic` array** and renders through a different path ‚Äî it does NOT get the `cage-extraregion` class. So scoping Easy Shade to `path.cage-extraregion` is safe: it never colours non-constraint decoration. (`window.convertedPuzzle` is the handy decoded-puzzle global; the `/api/puzzle/<id>` response is `fpuz‚Ä¶` = LZString-compressed f-puzzles, not directly readable.)
- ‚úÖ **Cell membership of a `#cages` shading path ‚Üí `SVGGeometryElement.isPointInFill` at each cell centre** (`pt.x = c*cs + cs/2`). `#svgrenderer`/`#cages`/the path carry no element transform, so cell-grid user units map straight into `isPointInFill`'s local coordinate space. This is how `assignExtraRegionColors` finds which cells each grey "extra region" covers (v2.123).
- ‚öÝÔ∏è **Don't derive cell size from `cage-box` path coords.** SudokuPad draws each box outline with a *midpoint* vertex at the box centre ‚Äî e.g. a 3√ó3 box of 64px cells emits `M0 0 L96 0 L192 0 ‚Ä¶` (96 = box centre, not a cell boundary). GCD-of-coords on that gives 96 and makes a 9√ó9 look like 6√ó6. Use `getGridCellSize()` (GCD of the `path.cell-grid` coords, with `dataset.spdrOrigD` fallback since our z-order fix clears `d`) + `detectGridSize()` instead. `.cell` element count also confirms grid size (81 = 9√ó9).
- ‚úÖ **Greedy 4-colouring, highest-degree-first** gives a valid adjacency-aware colouring for the grey-region touch graph (planar). Same approach as `computeRegion4Colors`; a 5th-colour fallback (reuse least-clashing) guards the rare non-planar case.
- ‚öÝÔ∏è **`drawRegionSplitBorders` is called by observers decoupled from `applySettings`** (the `#cell-grids` observer fires when we clear the cell-grid `d`; the position observer; etc.). So anything it draws must not depend on state set earlier *in applySettings only* ‚Äî e.g. the shaded-region clones recompute `assignExtraRegionColors(svg)` inside the `needShaded` block, because an observer-triggered draw would otherwise read stale/cleared `dataset.spdrExtraColorIdx` and produce **0 clones** (regions invisible). Symptom we hit: idx correctly set, originals hidden, but the shaded `<g>` empty.
- ‚úÖ **Region full-cell fill = exact cell rects, no outward growth** (v2.124). Growing each boundary side by SW to tuck under the strips caused the fill to poke past the puzzle's outer border (no covering strip there) and to overlap neighbours into a lighter double-painted seam at interior boundaries. Plain `c*cs, r*cs, cs, cs` rects fixed both; strips still draw on top at the boundaries with no visible gap.

## Cascade / load-order

- ‚ùå **Don't try to keep our `<style>` last in `<head>`** (tried v2.124 `startStyleTagGuard`, reverted v2.125). DarkReader inserts a `darkreader--sync` `<style>` immediately **after every** `<style>` it manages ‚Äî *including ours* ‚Äî so "be last" is unwinnable: DR just re-syncs after us. A `<head>` observer that re-appends ours fights DR's observer and only causes thrash. The colour-swatch black-on-load is a genuine DR load-timing race (a reload clears it); the `--cell-color` specificity bump (`html[data-darkreader-scheme="dark"][data-darkreader-scheme]`, (0,2,1)) helps but isn't a guaranteed fix.
- ‚úÖ **Colouring our injected UI consistently under DR ‚Äî what finally worked (v2.127): a LITERAL colour + `!important` + strip `data-darkreader-inline-*` once.** Once you set an `!important` inline colour and remove DR's marker, DR does **not** come back to re-override it (verified: stable for 300 ms+, no observer needed for the steady state). Re-assert on the element's "appear" moment (card ‚Üí `showCard`; button ‚Üí its DR observer / `applyToggleStyle`) to cover the load-time pass. The theme purple is the literal `rgb(181,104,228)` (== DR's converted `--controls-button-bg`).
- ‚ùå **Dead ends for the same problem** (all caused the white‚Üîpurple flicker, removed):
  - **Snapshotting a computed colour** (`getComputedStyle(undo).color`) at build time ‚Äî races DR's build-time conversion, captures grey or purple at random.
  - **Inline `var(--controls-button-bg)`** ‚Äî DR rewrites var usage only inside *stylesheets*, not inline, so it resolves to the raw dark *background* purple `#6a1b9a`. Even `var(--darkreader-text--controls-button-bg, ‚Ä¶)` was intermittent (that DR var isn't always defined when our inline style is first evaluated).
  - **A stylesheet rule** with `var(--controls-button-bg)` added late (not in DR's initial scan) ‚Äî DR didn't rewrite it either; resolved dark.
- ‚öÝÔ∏è **DR converts CHILD text nodes independently of the parent.** Forcing `color` on the button is not enough ‚Äî the child `<div>` label (and mode indicator) go grey on their own. Force the colour on every text element (loop over `[btn, lbl, modeLbl]`; the card loops over `cardTextEls`), and observe the button **subtree** for DR markers, not just the button.
- ‚úÖ **Swatches** (the 4 palette squares) still use literal `!important` + strip + the same subtree observer ‚Äî they must keep their exact palette colours, not be DR-converted.
- ‚úÖ **Action buttons (Fill/Clear/Clear All) migrated to the literal too (v2.129).** They used to snapshot `colorRefStyle.color` *and* `syncClipperOffsets` re-applied a live re-read ‚Äî both the snapshot race and a DR ping-pong on hover (DR re-scans when the label text/size changes on expand). Now `textColor` is the literal `rgb(181,104,228)` + `watchDR`, and `syncClipperOffsets` no longer touches the colour (only geometry/bg).

## Border-strip drawing

- ‚úÖ **Filled rects, one per contiguous run** of cells along an edge (not one rect per cell, not `<line>`s). Horizontal strips are trimmed by SW at ends where a perpendicular vertical strip exists; vertical strips own the corner pixels; concave inner corners (3 of 4 surrounding cells in-region) get an SW√óSW patch.
- ‚ùå **Dead end ‚Äî overlapping `<line>` + `<clipPath>`:** square line caps double-painted at every internal joint and at perpendicular corners ‚Üí visible darkening at <100% opacity. Replaced by the filled-rect approach (also dropped all `<defs>` / `<clipPath>`).
- ‚ùå **Bug (fixed) ‚Äî forgetting `parseFloat` on stripe width:** `settings.regionColorStripeWidth` is a string (`"3"`) from the `<input>`. Without `parseFloat`, `x + SW` string-concatenates ‚Üí rects at x‚âà6415 off-board ‚Üí SudokuPad shrinks the board to a 9√ó9 miniature. Always `parseFloat(...) || 3`.

## Kropki detection

The trap: SudokuPad uses the same `rect.textbg` class for Kropki circles **and** non-Kropki labelled circles (arrow constraints, operators).

- ‚úÖ **Robust detection = THREE independent signals, all required (v2.184).** A Kropki-type edge dot is: (1) **shape** = a true circle; (2) **fill** = pure black `#000000` or pure white `#FFFFFF`; (3) **position** = centre on a cell border (`isOnCellBorder`). `isKropkiCircle` = signal 1 only; `isKropkiDotRect(rect,cs)` = 1‚àß2‚àß3 and is the single source of truth for "the colour fix owns this" (object shading + label-bg both skip `isKropkiDotRect`, so a dot is never double-claimed). The three signals were hardened together in v2.184 after three separate puzzles each exposed one gap:
  - ‚ùå **`rx ‚âà w/2` alone does NOT prove "circle" ‚Äî it also matches a PILL's short side (v2.184 fix for issue #2).** A vertical 2-cell double-arrow bulb is a `textbg` rounded rect w=48,h=112,rx=24; `rx ‚âà w/2` (24‚âà24) passed, so it was mis-detected as Kropki, force-coloured white + given a `~`. (Horizontal pills w=112,h=48,rx=24 already failed since rx‚âÝw/2 ‚Äî that asymmetry is why only *some* bulbs broke.) Fix: a circle requires **`w ‚âà h` AND `rx ‚âà w/2`**. Regression puzzle `bill-murphy/20260526-david-and-goliath-sudoku`.
  - ‚ùå **Requiring the `feature-kropki`/`textbg` class missed CLASS-LESS dot circles (v2.184 fix for issue #3).** clover's "Diamond Ring" (`nipb00tmn1`) draws its white edge dots as bare `<rect>` circles in `#overlay` with **no class at all**. They failed `isKropkiCircle` (class gate) ‚Üí never colour-fixed ‚Üí DR left them dark; *worse*, they matched the white-fill arm of `LABEL_RECT_SEL` so `fixLabelRect` painted them the dark label-bg colour (the "black dot"), and the black stroke DR-inverted to white (the "white circle" ring), with no `~` injected (not detected as a bare dot). Fix: `isKropkiCircle` now accepts a geometric circle whose class is `textbg` **or absent** (rejects other *named* classes); the scans (`fixAllKropkiDots`/`rebuildKropkiLabels`/`HT.kropki`) widened to `‚Ä¶ , #overlay rect, #underlay rect`, with the fill+position gates doing the real filtering. **Safe because the position gate (signal 3) is the discriminator** ‚Äî this is the v2.164 philosophy, NOT a repeat of the reverted v2.166 "force-keep author white" (which had *no* positional signal). Verified 0 newly-grabbed / 0 dropped on both test puzzles before shipping. SudokuPad never emits `<circle>` (all rounded rects), so rect-only scanning is complete.
  - ‚ùå **Object shading and label-bg must SKIP Kropki dots, or a black dot reads as a "gray object" (v2.184 fix for issue #1).** `shouldShadeOverlayRect` only skipped grayscale **`textbg`** dots, so black **`feature-kropki`** dots (e.g. `264wvenhmu`, native Kropki in `#overlay`) fell through ‚Üí counted as gray objects by `objFillSources`, so the **Gray object brightness** eyeball highlighted them (white dots were excluded only incidentally, by the near-white rule ‚Äî hence the black/white asymmetry the user noticed). The render was correct only because `fixAllKropkiDots` runs *after* `fixAllUnderlays` and overwrites. Fix: `shouldShadeOverlayRect`, the `#underlay` arms of `fixAllUnderlays`/`objFillSources`, the observer's underlay branch, and `fixLabelRect`/`HT.labelBg` all bail on `isKropkiDotRect` ‚Äî so apply + highlight agree (foolproof principle) and the Kropki section is the dots' sole controller (Kropki off ‚áí they fall to DR, not to object shading).
- ‚úÖ **Disambiguation rule = POSITION only (`isOnCellBorder`), v2.164.** A circle (black *or* white) is a Kropki-type dot **iff its centre sits on a cell border** ‚Äî a gridline in one axis, mid-cell in the other. That's the defining geometry of any edge clue (Kropki / X-V / operator dots like Makodoku `+`/`√ó`). The colour fix (`fixKropkiDot`), label injection (`rebuildKropkiLabels`), **and** the `HT.kropki` highlight all gate on it (foolproof principle). Off-border circles are left to DarkReader: **quadruples** sit on a grid **corner** (gridline in *both* axes), arrow bulbs / cosmetic circles at a cell **centre**, line endpoints wherever a path ends.
  - ‚ùå **The old v2.137 OR-clause `svgHasBlackKropkiCircle` was the bug (removed v2.164).** It said "if the puzzle has *any* black Kropki dot, treat *all* white circles as Kropki" ‚Äî so on a puzzle with both a black dot and a quadruple (test puzzle `pbwqsppuho`), the quadruple's white corner-circle got force-coloured **and** a `~` label drawn over its existing 1-4 digits. Position alone suffices: Makodoku's all-white border dots still pass `isOnCellBorder` (which is why the black-dot anchor became redundant once the border test was added in v2.137). Note `getKropkiAdjacentText` did *not* catch the quadruple's digits (they're not an immediate text sibling), so the no-text gate alone didn't save it ‚Äî the position gate does.
- ‚úÖ **Two-tier shape test (each then gated by `isOnCellBorder`):** `isKropkiCircle` (shape) ‚Üí colour fix; `isKropkiRect` (shape *and* no text sibling) ‚Üí label injection. Labelled Kropki circles (Difference / Ratio / Makodoku operators) get their colour fixed but are **not** re-labelled, and the no-text-sibling gate also means the rotate-on-horizontal-border setting never touches them.
- ‚ùå **Naively forcing _all_ white circles white does NOT work:** it inverts arrow-constraint puzzles (white circles carrying arrow glyphs like ‚Üñ, sitting at cell centres) into black-outlined shapes, overriding DarkReader's correct dark-mode rendering. (The bug behind the v2.119 fix; regression puzzle `jhrb0vsbnk` ‚Äî verified still off-border/untouched under the v2.137 edge rule.)

## Lines (thermos, palindromes, renban, whispers, arrows‚Ä¶)

- Every line-type clue renders as a stroked `<path>` in `#arrows` (`fill=none`). A thermometer is **two separate elements**: the **bulb** is a rounded `rect` in `#underlay` (caught by Object shading as a *fill*), and the **shaft** is one of these `#arrows` paths. They share a source colour (e.g. `#CFCFCF`). DR leaves the line near-white in dark mode ‚Üí a mismatch against shaded fills.
- ‚öÝÔ∏è **X-diagonals (`sudokuX`) live in `#overlay`, not `#arrows`** ‚Äî a stroked `<path>` with its own colour (e.g. `#34BBE6`). `fixAllLines` is scoped to `#arrows`, so it does **not** shade diagonals: they keep their native colour and are left to DR. Fine for a saturated diagonal (reads clearly on dark); a near-grey diagonal could in theory stay too bright under object-shading, but not observed. Confirmed on test puzzle 2 `3x3zm2co6o`. Targeting just diagonals is fragile (no distinguishing class), so deliberately left alone.
- ‚öÝÔ∏è **Line ENDPOINT circles** (between/lockout lines) are `rect`s in `#overlay`, not `#arrows`. Authors give them different alphas on purpose: solid bulbs `#FFFFFF`, see-through bulbs `#ffffff80` (so the line reads *through* the disc). Both match `LABEL_RECT_SEL`'s case-insensitive white-prefix arm (v2.196) and are darkened by the label-bg fix; coloured ones (e.g. diamond `#e7e6ff80`) go through object-shading instead. **`fixLabelRect` preserves each rect's authored alpha** (v3.8.0: `applied = labelBgOpacity √ó srcAlpha`) so `#ffffff80` stays at 0.5 and the line shows through ‚Äî matching DR (DR darkens a fill while keeping its alpha). Without that, every white-prefixed rect flattened to one opaque colour and translucent bulbs wrongly blocked what was under them. Confirmed on `pbwqsppuho` (double-arrow bulbs solid, between-line bulbs see-through). **DR's natural per-bulb difference is just the author's alpha, NOT a z-level thing** ‚Äî all four bulbs are in the same `#overlay` layer.
- ‚úÖ **Shade the line stroke** with the same Object-shading transform as fills (`computeObjectShade`, applied to the stroke in `applyLineStroke`). Gray lines follow the linked Gray slider, colored lines the Brightness/Opacity sliders. Stroke **width is never touched** ‚Äî only stroke colour + opacity.
- ‚öÝÔ∏è **Scope is now ALL stroked `#arrows` paths** (v2.140, `fixAllLines`/`isLineStroke`). The old v2.122 rule matched only shafts whose stroke equalled a `#underlay` bulb fill colour (`getBulbFillColors`/`isThermoShaft`, removed) ‚Äî so **bulbless** line puzzles (palindromes/quads like `9p6eahqmux`, which have *zero* `#underlay` rects) fell through to DR and stayed bright. Broadening to every stroked `#arrows` path fixes those and is intentional: real arrow-sudoku arrows are shaded too now (user wanted it; dial back here if it ever regresses).
- DR converts the stroke via `data-darkreader-inline-stroke` (not `-fill`); the SVG observer watches that attribute too. Our inline `stroke ‚Ä¶ !important` overrides DR's stylesheet `!important` rule; no mutation loop observed.

## Gray vs colored object shading (v2.140)

- ‚úÖ **Gray and colored need separate sliders.** `shadingTransform` forces **full saturation** for colored objects (vivid at L=0.4) but returns plain `rgb(L¬∑255 ‚Ä¶)` for gray (saturation 0) ‚Äî genuinely dark at the same number. So one slider can't serve both. Each side has its own brightness + opacity keys (color `underlayLightness`/`underlayOpacity` @ 0.4, gray `underlayGrayBrightness`/`underlayGrayOpacity` @ 0.6). `computeObjectShade(c)` routes on `isGrayColor` (saturation `< 0.08`) and returns `null` when the relevant control is disabled (caller then clears its inline style ‚Üí back to DR). Used by both fills (`applyShadingFill`) and lines (`applyLineStroke`).
- ‚úÖ **Combined vs separate is UI-only (v2.141).** The **"Control opacity and brightness separately"** checkbox (`underlaySeparateBrightnessOpacity`, default off) just swaps which rows show: off ‚Üí one *combined* slider per side that writes both axis keys to the same value (`makeRangeRow` `extraKeys`/`extraEnabledKeys`); on ‚Üí the two axes as separate sliders. The shading code never reads the flag ‚Äî it always reads the 4 keys, which combined mode merely keeps equal. Entering combined mode forces `opacity := brightness` (both sides) so the single slider matches what's applied.
- ‚öÝÔ∏è **Several rows share a setting key** (combined-color and separate-color-brightness both bind `underlayLightness`), so `makeRangeRow`'s per-row `controlSyncers[key]` registration clobbers. Fix: the object-shading `subBuilder` overrides `controlSyncers` for all its keys with one `refreshAll` that re-syncs **every** row (via each row's `spdrRefresh`) and re-applies the visible-row mode ‚Äî so both the section ‚Ü∫ and the bottom "Reset all" restore the UI correctly.
- Shape **outlines** (`applyShapeStroke`) follow the same combined/separate toggle as colored/gray (Border brightness + Border opacity, `underlayStrokeLightness`/`underlayStrokeOpacity` @ 0.75). Brightness preserves the element's hue via `shadingTransform`; opacity became configurable in v2.143 (was hardcoded `stroke-opacity: 1`).
- The section ‚Ü∫ deliberately **omits** `underlaySeparateBrightnessOpacity` from `resetKeys`, so resetting a section's values doesn't flip the combined/separate checkbox; only the bottom "Reset all" (which copies all of DEFAULTS) restores it.
- ‚úÖ **Coloured cosmetic shapes keep their colour; only white/grey label boxes darken (v2.165).** `rect.textbg` is shared by Kropki dots, white label boxes (cage sums / little-killer / XV operators) AND author-**coloured** cosmetic shapes (e.g. a light-blue `#80E5FF` circle or `#5EE7FF` box behind a digit). `fixLabelRect` used to flatten *all* matches to the dark label colour, so coloured cosmetic shapes rendered **black**. Fix: `fixLabelRect` skips SATURATED fills (`!isGrayColor`), and `shouldShadeOverlayRect` now **shades** coloured `textbg` (still skips white/near-white + grayscale textbg = Kropki/grey labels). So coloured cosmetic shapes route through Object shading (dimmed, hue preserved) like any other overlay shape; white/grey label boxes still darken. `objFillSources` reuses `shouldShadeOverlayRect` so the `objColored` highlight follows automatically; `HT.labelBg` filters out the coloured ones to match `fixLabelRect`. Found on test puzzle 2 `3x3zm2co6o`.
- ‚úÖ **`fill="none"` (or transparent) label rects are INVISIBLE anchors ‚Äî never darken them (v2.176).** Authors reuse a `rect.textbg` purely to position a label (e.g. a little-killer "12" outside the grid sitting *over* an arrow). Its author fill is `none`, so it renders nothing and DR ignores it (no `data-darkreader-inline-fill`). `fixLabelRect` was painting it our `#181A1B` label-bg `!important` ‚Üí an opaque dark box that **covered the arrow beneath it**. The tell that it's our doing, not DR's: turning the Label-background checkbox off (`removeProperty('fill')` ‚Üí reverts to `fill="none"`) made the box transparent and the arrow reappear; and the rect carried **no** DR marker/var. This is also why these differ from the white cage/little-killer label boxes (`fill="#FFFFFF"`), which DO need darkening ‚Äî same `textbg` class, different fill attribute (`none` vs white). **Label backgrounds are darkened by TWO mechanisms ‚Äî both must skip `none`:** (a) inline `fixLabelRect` (skip when raw fill attr is `none` or parses `a===0`; `parseColor("none")` returns `null` like an absent attr, so test the **raw attribute string** `=== 'none'`), AND (b) an injected **CSS rule** in `buildCSS` (`rect.textbg { fill: <bg> !important }`). Fixing only (a) made the box go **light grey `#D8D5CF`** instead of dark, not transparent: DarkReader **rewrites our injected CSS rule** (sees the dark `#181A1B` fill in a stylesheet ‚Üí inverts it to light), so with the inline gone the DR-lightened CSS rule took over. (The inline `!important` had been overriding DR's counter-rule ‚Äî the "inline beats DR, stylesheet loses to DR" lesson in action.) So the CSS selector also needs `rect.textbg:not([fill="none"])` (both the dark-scheme and purple-theme copies). With neither mechanism touching it, the rect falls back to its `fill="none"` attribute ‚Üí transparent ‚Üí arrow shows. (8 such rects on test puzzle `pbwqsppuho`.)
- ‚ùå **Don't force-keep author white on bare cosmetic shapes (tried v2.166, reverted v2.167).** A throwaway test puzzle had a white cosmetic circle (class-less white rect) that DR darkened to near-black; we briefly added `isLineEndpointCircle` (geometry: is the circle within ~1 radius of an `#arrows` line end?) to tell standalone cosmetic circles from between-line endpoints, then force-kept `#ffffff` on the cosmetic ones. **Reverted: narrow in intent but BROAD in reach** ‚Äî it force-whitened *every* class-less white shape on *every* puzzle, risking regressions across the ~100 already-vetted puzzles (a previously-fine darkened white shape would suddenly glare white). Deeper point: **there is no broad signal** that distinguishes "author wants this white" from "fine to darken" ‚Äî DR's broad answer is *darken*, so preserving white inherently needs a per-anomaly heuristic or an explicit user toggle, neither worth it for a one-off. **Conclusion:** bare white cosmetic shapes follow the broad rule and **darken** like any white non-underlay rect. (The v2.165 COLOURED-shape fix stays ‚Äî it's broad and correct: coloured cosmetics route through Object shading with hue preserved.)

## Settings-clarity highlight overlay (v2.154‚Äìv2.163)

The üëÅ hover-icons that show "what does this control affect" on the board. Architecture + the gotchas that took several iterations:

- ‚úÖ **FOOLPROOF PRINCIPLE ‚Äî a control's highlight must enumerate the SAME elements via the SAME predicates its apply code touches**, never a hand-written parallel selector (those drift and were wrong repeatedly). Object-shading getters reuse `objFillSources`/`objLineSources` (= `fixAllUnderlays`+`fixAllCagePaths`+`fixAllLines`) and route via `computeObjectShade` rules; `labelBg`=`LABEL_RECT_SEL`, `kropki`=`isKropkiCircle` filter. Consequence to remember: **lines route under the colored/gray sliders (by stroke colour), and "Border" = shape OUTLINES only** ‚Äî not lines. And labelBg legitimately overlaps kropki on the circular `textbg` dots (label-bg recolours those too) ‚Äî that overlap is correct, not a bug.
- ‚úÖ **Trace real geometry, never a bounding box.** A single `<path>` (e.g. the whole grid) has a bbox = the whole board, so a bbox outline looked like just the outer square. Clone the actual element into an overlay `<svg>` and stroke it.
- ‚úÖ **Use `el.getScreenCTM()`, NOT `getCTM()`.** getCTM maps to the board's viewBox space ‚Üí fragile to match. getScreenCTM maps element-local ‚Üí CSS px; put clones in a full-window overlay svg (`viewBox 0 0 innerW innerH`, `preserveAspectRatio none`) and they land exactly over the original regardless of viewBox/scale/scroll. Map raw points with the same matrix `{x:a¬∑ux+c¬∑uy+e, y:b¬∑ux+d¬∑uy+f}`.
- ‚úÖ **De-duping adjacent shaded cells = occupied-cell-SET perimeter, NOT edge-counting.** Shaded cells are small INSET rects, so outlining each drew parallel lines between neighbours. Edge-count-and-drop-if-2 *seemed* right but broke when a cell holds multiple shaded elements (underlay tint + overlay rect): a perimeter edge counted twice got dropped, interior lines survived. Correct approach: build the set of occupied grid cells (snap each axis-aligned rect to its cell via `getGridCellSize`), then draw an edge only where the neighbour cell is absent. Robust to N elements/cell. Rotated rects (diamonds) + non-rects trace individually.
- ‚úÖ **Paint modes** (`styleNode`): default **stroke** (outline) is what the user wanted for objects/selection ‚Äî a translucent **fill** overlay read as "solid shading over the object" and was rejected *except* for thin shapes: `PAINT.regMulti='fill'` (multi-color border strips are 4px rects ‚Üí fill = one clean bar, outline = doubled lines). `PAINT.cellColors='bbox'` outlines the cell square so a 2-colour cell's diagonal split isn't traced (the diagonal = top half of the colour boundary).
- ‚úÖ **Click = pulse the highlight, not toggle the effect.** The wanted click behaviour is a gradual brighten‚Üídim of the *highlight outline* (`spdrHi.pulse`, `@keyframes spdrHiPulse` ending at full opacity so it coexists with the steady hover-hold). An earlier on/off toggle of the real setting (`blinkEffect`) was wrong ‚Äî removed.
- ‚úÖ **Icons stay usable in a disabled section** via `pointer-events:auto` on the icon (a child re-enables pointer events under the section's `pointer-events:none` dim) ‚Äî lets the user preview before enabling. Caveat: sections using `collapseWhenDisabled` (Region borders) hide their sub-rows entirely, so those icons vanish when off ‚Äî not yet addressed.
- ‚úÖ **Hover "examples" are safe because they're overlay-only or transient UI.** Pencilmark examples (`drawPencilExample`) draw demo digits into the overlay svg (`spdrHi.addText`) ‚Äî they NEVER touch the puzzle model, so zero risk. The selection example is the one real-state action: `app.select(cells)` (selection is transient UI, intentionally left selected). Writing real pencilmarks via the app was avoided (would pollute solve/undo).
- **DOM facts (this codebase):** board svg = `#svgrenderer` (it *is* the `<svg>`); tool buttons `#control-colour`/`#control-centre`/`#control-corner`; cells are 64px full-cell `#underlay` rects but object-shading decorative squares are inset `#overlay` rects; multi-color region borders = many thin strip rects marked `data-spdr-kind="multi"` (center=path, cell=grid clone). Native `#cell-colors` opacity = CSS `var(--cell-color-opacity)` = **0.6** (so `cellColorsOpacity` default is 0.6 ‚Äî enabling at default is a no-op).

## Centering an injected glyph on a Kropki dot (v2.183 ‚Äî alphabetic baseline + geometricPrecision; supersedes v2.175 raster + v2.181 middle-baseline)

Centering the `:`/`~`/custom label *visibly* inside its circle is harder than it looks ‚Äî `text-anchor:middle` + a baseline only center the glyph's **line box**, not its visible **ink**, and the gap differs per glyph. **There is no reliable RUNTIME way to measure ink here** ‚Äî every measurement path has a fatal flaw, so do not try again:

- ‚ùå **`getBBox` / `getBoundingClientRect`** return the line/cell box (~19px tall for a `~` at font-size 16), not the ink. They report the label "perfectly centered" while it visibly isn't.
- ‚ùå **Canvas `measureText` `actualBoundingBox*`** metrics don't match SudokuPad's SVG rasterizer ‚Äî the SVG renders the same glyph ~2px taller/offset. (That's the *metrics* API. Rasterizing with `ctx.fillText` + a `getImageData` pixel-scan ‚Äî used for one-time calibration, below ‚Äî IS reliable; it renders the real font outline. The two are not the same thing.)
- ‚ùå **`<img>`-isolated SVG** (clone the label into a standalone `<svg>`, rasterize via `new Image()` ‚Üí canvas) **falls back to a different font** than the page (notably for `~`), so the measured ink lands in the wrong place.
- ‚ùå **The v2.175 "measure the actual render" approach** built on that isolated-SVG raster, cached per font signature and re-applied async. **Removed v2.181 ‚Äî two fatal problems:** (a) `rebuildKropkiLabels` re-runs on *every* SudokuPad node insertion, so the async result raced the rebuild; (b) when measurement went wrong it didn't fail small ‚Äî in v2.180 it **flung the colon to dx‚âà‚àí12, dy‚âà‚àí11 on a ~21px dot**, i.e. clean out of its circle (the exact "symbols aren't even in the circle" bug). Pixel-perfect *automatic* centering of arbitrary glyphs is not achievable here.
- ‚úÖ **What works: a FIXED per-glyph nudge off the ALPHABETIC baseline, applied synchronously** (`centerKropkiLabel` + `KROPKI_INK_NUDGE`). Force `dominant-baseline:alphabetic` inline-`!important` on the label (beats SudokuPad's `middle` CSS rule, which a presentation attribute can't), then set the anchor to the circle center plus `{dx,dy}` in **fractions of font-size**. No measurement, no async, no cache ‚Üí rebuild-safe by construction.
  - **Why the alphabetic baseline ‚Äî the v2.182 CROSS-BROWSER fix.** v2.181 centered off the `middle` baseline; it looked right in **Blink (Chrome/Brave)** but **Gecko (Firefox/LibreWolf) shifted the ROTATED glyphs left.** `dominant-baseline:middle`/`central` require a font-metric computation (x-height / em-center) that the two engines do slightly differently ‚Äî an unnoticed *vertical* difference upright, but the `rotate(90,cx,cy)` on horizontal-border dots maps the vertical axis onto the horizontal one, turning it into a visible *horizontal* shift (Gecko only). The **alphabetic baseline needs no metric computation ‚Äî every engine agrees on it** ‚Äî and a glyph's ink height above it is a pure font-outline property (same font everywhere), so a fraction calibrated in one browser centers it in all of them, both orientations. (`text-anchor:middle`, used for horizontal centering, is *also* engine-consistent ‚Äî which is why only the baseline/rotated axis ever broke.)
  - **Why geometricPrecision ‚Äî the v2.183 PER-DOT + cross-engine fix.** At the small label size (~16px) the font is **hinted**: each glyph's outline is grid-fitted toward whole device pixels, bending its ink ~0.5px sideways / ~0.7px down off true geometry ‚Äî by an amount that depends on the dot's **sub-pixel position** (so every dot drifts differently; ~0.8px vertical spread measured across phases) AND on the **engine's hinter** (so Blink ‚âÝ Gecko). The vector Kropki circle behind it is NOT hinted, so the glyph slides off-center, inconsistently ‚Äî the "even within one browser they aren't consistent" symptom. Forcing **`text-rendering:geometricPrecision` inline-`!important`** makes the live SVG render the true unhinted outline ‚Äî identical on every dot and every engine. No other lever disables hinting: `optimizeSpeed`, rendering a large glyph and scaling it down, etc. were all tested and do **not** help.
  - ‚öÝÔ∏è **geometricPrecision is honored by the LIVE inline-SVG paint, NOT by an `<img>`-decoded SVG.** A serialized `<svg>` ‚Üí `new Image()` ‚Üí canvas test shows `auto == geometricPrecision` and is **misleading** ‚Äî that rasterization path ignores `text-rendering`. The real board (inline SVG painted in the document) *does* apply it. Verify the effect on the live board / with the user, never via an img-raster proxy. (This wrong proxy cost a cycle in v2.183 ‚Äî I nearly concluded geometricPrecision was a no-op.)
  - **Calibrated values = ink-center height above the baseline, measured at LARGE size** (‚â•400px, where hinting vanishes ‚Üí the same unhinted geometry geometricPrecision renders): `:` ‚Üí `dy 0.275`, `~` ‚Üí `dy 0.324`, unknown glyphs `dy 0.35`; all `dx 0` (the hinted ~0.5px left-shift is a hinting artifact geometricPrecision removes). `cy + dy¬∑fs` drops the baseline below center so the ink lands on center. **Tweak a value if a glyph looks off ‚Äî that's the intended knob.**
- ‚öÝÔ∏è **Rotation works in every engine *because* centering no longer depends on an engine-specific baseline.** Apply the nudge in the **unrotated** frame (set `x/y`; `rotate(90,cx,cy)` pivots around the circle center). Ink centered on `(cx,cy)` stays centered under rotation about `(cx,cy)`, so one nudge serves both orientations. The v2.181 rotation bug was never the rotation math ‚Äî it was the baseline divergence the rotation merely *exposed*.
- üîç **Calibrating without screenshots (the reliable way).** Measure the ink via **canvas**: `ctx.font='600 <fs>px Tahoma, ‚Ä¶'; ctx.textBaseline='alphabetic'; ctx.textAlign='center'; ctx.fillText(glyph, ax, ay)`, then scan `getImageData` for the ink bbox. Canvas `textBaseline:'alphabetic'` **is** the SVG alphabetic baseline, and it rasterizes the real page font, so `(ay ‚àí inkCenterY)/fs` is exactly the `dy` to use ‚Äî and it's browser-independent (pure font outline), so one measurement serves all engines. Big `fs` (e.g. 400) for precision. (Bonus: the CDP/`Page.captureScreenshot` pipeline wedged mid-session ‚Äî canvas needs no screenshots at all. Don't resurrect the isolated-`<svg>`‚Üí`<img>` raster; it had the font-fallback bug. Visual cross-check, when screenshots work: draw an isolated `position:fixed` overlay with the glyph big at `dominant-baseline:alphabetic` + a pixel ruler; the `zoom` tool crops 1:1 so the glyph must be physically big.)
