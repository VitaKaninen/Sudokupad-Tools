# Lessons Learned — SudokuPad DarkReader Fix

*Flat list of what's confirmed to work, what's confirmed NOT to work, and why. The point of this file: never re-fight a solved battle. When a session confirms a new fix or a dead end, add a line. Record the conclusion, not the journey.*

## Native dark mode — the substrate swap (v2.195, in progress on branch `native-mode`)

**Direction change:** stop fighting DarkReader; evict it from SudokuPad and ride the site's own native dark mode instead. DR is a blind, adversarial colour *inverter* that re-runs on every mutation; native dark mode is the author's *semantic*, *static* stylesheet. Below "Beating DarkReader" describes the OLD architecture (the `main` branch). This section is the new one.

- ✅ **`<meta name="darkreader-lock">` evicts DR cleanly — confirmed in the user's DR build.** Injected *dynamically* at runtime, DR tore itself down within ~1.2 s: `data-darkreader-scheme` attr removed, **all** `style.darkreader` tags removed (43 → 0), `--darkreader-*` vars gone. The leftover `data-darkreader-inline-fill` *attributes* on elements become inert orphans (no rule references them). Our top-of-IIFE `lockDRUseNative()` block injects the meta at `document-start` (head may not exist yet → MutationObserver fallback). DR keeps working on every other site. (NB a DR regression broke the lock in build v4.9.86 — verify per-environment; the user's build works.)
- ✅ **Native dark mode = the `darkmode` flag inside the `svencodes_settings` localStorage object → `body.setting-darkmode` class.** Our block sets the flag true (so SudokuPad's own toggle reflects it + re-applies on init) AND adds the class live (no reload / load-order dependence). **Side effect: this persistently flips the user's SudokuPad `darkmode` setting on** (reversible in SudokuPad settings).
- **Native palette** (`.setting-darkmode` CSS vars): `--dm-black:#1a1a1a`, `--dm-white:#eee`, `--dm-button-color:#8522c3` (purple), `--dm-rulesbg:#265016`, `--dm-userblue:#5f95ec`, `--dm-black-alpha:rgba(18,18,18,.7)`.
- **What native dark mode HANDLES (for free, correctly):** `.cell-grid`/`.cage-box` strokes → `--dm-white`; white labels via `rect.textbg[fill="#FFFFFF"]` (+ case variants) & `.textbg_ffffff` → `--dm-black`; `[bordercolor="#fff"…]`; givens (`.cell-given`/`.givenCornermark`/`.cell-candidate .given`) → `--dm-white`; `.dialog`, `#controls` text, `.puzzle-rules`, settings toggles/labels, `#fog-fogcover`.
- ⚠️ **What native dark mode MISSES → the 3 known gaps the rest of the script must fix:** (1) **arbitrary gray / translucent-dark objects aren't inverted** — cell-shading authored as translucent black (`#0003`) and dark grays stay dark → vanish on the dark page (DR *did* invert these). → our **Object Shading** covers it, and is *more* needed here, not less. (2) **white labels that aren't an exact `[fill="#FFFFFF"]` match** — translucent-white `#ffffff80` endpoints, class-less white circles → stay white, white text unreadable. → our **label-bg fix** is a broader white/near-white match (currently its native CSS branch keys on `body.setting-uitheme-purple`; re-key to `setting-darkmode`). (3) **control buttons stay bright** — native themes `#controls` *text* only; `#control-settings`/`-fullscreen`/`-rules` keep `background:#eee`. → needs one small new darkening rule.
- **Boot gate moved off DR:** the script used to gate its whole startup on `isDarkReader()` (`data-darkreader-scheme="dark"`). Now `isDarkMode()` = `isNativeDark()` (`body.setting-darkmode`) ‖ `isDarkReader()`, so it boots on native and still works if DR somehow remains. `waitForDRAndSVG` also observes the body `class` attr (native lands there).
- **Migration shape (still TODO):** re-base DR-sampled defaults (`#181A1B`/`#dddad6`/`#e8e6e3`) → native palette; re-key the label-bg CSS to `setting-darkmode`; generalize/remove the `inDR` branches in `fixCageBox` etc.; **amputate the entire `captureDrInline`/`restoreToDr`/`spdrDrFill` restore-to-DR machinery** (no DR var to restore to — on disable just clear our inline & let native CSS show); handle `--cell-color-*` swatches under native (the `[data-darkreader-scheme]` specificity-bump rule is DR-only).

## Native-mode FROZEN_DARK_CSS swap gotchas (v3.24–3.26 + one open item)

The blanket `#000↔#fff` / `#fff→dm-black` attribute swaps in `FROZEN_DARK_CSS` are powerful but blind — things they got wrong (the given-candidate + selection ones were reworked in v3.26 — see each bullet):

- ✅ **`<mask>` contents are luminance, not colour — never swap them.** The Restart toolbar icon is a circular-refresh arrow with the "!" punched out via an SVG `<mask>` (`rect fill="#fff"` = show-all, `path fill="#000"`/`stroke="#000"` = the "!" cutout). Our `[fill="#fff"]→dm-black` + `[fill="#000"]→dm-white` swaps **inverted the mask** (rect→black=hide-all, "!"→white=show) so only the "!" rendered and the arrow vanished. Fix: restore authored fill/stroke for any `mask [fill=…]`/`mask [stroke=…]` (`.spdr-dark mask [...]` — higher specificity than the bare swaps). Any icon using a `<mask>` with #000/#fff was being silently corrupted; this is strictly an improvement.
- ✅ **Given candidates invisible (black) — CSS-variable fix worked everywhere EXCEPT LibreWolf, so it was redone in JS (v3.26).** The verbatim-DMA block set `--puzzle-givenCandidate`/`--puzzle-givenCornermark` to `var(--body-bg)` (= `--dm-black`), so given pencilmarks rendered in the page-bg colour. v3.24 fixed it by changing the *variable* to `--dm-white` (the `.cell-candidate .given { fill }` selector loses on specificity to SudokuPad's `#svgrenderer .cell-candidate .given { fill: var(--puzzle-givenCandidate) }` — ID beats classes — so you must fix the variable, not pile on selectors). **That worked in Chrome/Brave/Firefox but NOT LibreWolf** (cause undiagnosed — no LibreWolf access; likely a custom-property cascade/timing quirk). **v3.26 the robust angle:** force valid given candidates white with an **inline `!important` fill** in `fixCenterTspan` (early-branch on `tspan.given`, always-on regardless of `centerEnabled`; conflicts `removeProperty('fill')` so SudokuPad's own pencilmark-error red still shows). Inline !important wins in every engine — don't rely on a custom-property override for a cross-browser visual. The `--puzzle-givenCandidate`/`-givenCornermark` var change is **kept** as a fallback (+ it covers cornermarks). (Test puzzle: `j8h4wznocm`, clover Pencilmark Sudoku.)
- ✅ **Cage-mode selection: native renders a grey block + a *duller* blue border; both are SudokuPad's, not ours.** Selection is `path.cage-selectioncage` with a hardcoded inline attr `fill="rgba(255,255,255,0.4)"` (grey on dark) and `stroke: var(--puzzle-selectioncage)` = `rgba(0,126,255,0.7)`. DarkReader 2.x **inverted the fill to a faint dark `rgba(24,26,27,0.4)` and brightened the stroke to `rgba(51,163,255,0.7)`** — that brighter-border/no-grey look is what users are used to. Native does neither by default. **v3.26 resolution:** the existing **Cell selection border** section (`selectionColorEnabled`, default now **on**) already restyles the cage stroke (`#cell-highlights path.cage-selectioncage { stroke: #3399ff@0.7; stroke-width:8 }` via buildCSS), giving the brighter border; a new sub-checkbox **"Remove gray background"** (`selectionHideBackground`, default on) adds `fill: transparent !important` in the same rule. So default = brighter border + no grey; **disabling the section reverts to the pure native selection** (rawer blue + grey fill). The old unconditional `.spdr-dark .cage-selectioncage { fill: transparent }` in FROZEN_DARK_CSS was removed (it can't be toggled). NB changing `selectionColorEnabled`'s default to `true` only helps fresh installs — existing users have it persisted `false` and must Reset-all or toggle once.
- ⏳ **OPEN — author-drawn `#overlay` grid frames whitened on top of our borders.** Authors draw a cosmetic grid border/structure as classless `#overlay path`, `fill:none`, `stroke="#000000"`, ~grid-sized; `#overlay` renders **above** our injected region-border group, and our `#000000→dm-white` swap whitens it into a bright frame sitting on our borders. Confirmed on `2s74gvrn07` + `5dv5v9gzux` (Bill Murphy — single full-grid square frame, clean case) and `24oouslvly` (clover "Dragon" — the *whole* grid + region lines are author overlay paths, not just a frame). Our region feature already clears the native `.cell-grid` and draws the grid via our clone, so **hiding the redundant frame loses nothing**. Catalog: `#overlay | path` = 37 puzzles, mixed stroke colours — a blanket "stop whitening black overlay paths" would also hide intentional cosmetic black lines. **Fix deferred** pending the user's full list of affected puzzles (to scope the detection rule). Detection signal that works: classless `#overlay path`, `fill:none`, grayscale-dark stroke, closed (`Z`) bbox ≈ board with `getTotalLength` ≈ 4×side (distinguishes a frame from a full-bbox diagonal line).

## Shaded "extra regions" rendered as grey underlay cells (v3.15–3.17)

Some puzzles author an extra region NOT with the Extra-Region tool (`#cages path.cage-extraregion`, the only thing the original Easy-Shade "Shaded" mode knew) but as a **hidden, sum-less, `unique` killer cage**, which SudokuPad renders as plain grey `#underlay rect`s — no path to clone. They're true regions (a repeat inside conflicts). Example: `zax289niwv` "We Live Here" (4 such regions, 36 grey cells + inset border strips, original fill `#0003`).

- 🔑 **What actually breaks init (the real cause, confirmed v3.17 by reading the source).** `Framework.getApp` is just `() => F.pApp` — returns a stored Promise — and `Framework.app` is a **side-effect-free getter** (`get(){ return __app; }`: `undefined` before the `App` is constructed, the `App` instance after). So *reading* the model **cannot** break init. What broke v3.15 was the **async `.then(…)` callback firing `applySettings()`** (a full re-apply) at an unpredictable point mid-load, re-entering the whole pipeline — kicked off from the hot per-rect paint path. The fragile thing is **async resolution + heavy re-apply during load**, NOT a guarded synchronous property read. (The earlier "never touch the model near load" rule was an over-correction.)
- ❌ **v3.15 (broken):** read regions via `Framework.getApp().then(app => …applySettings())`, kicked off from `extraRegionRectColor`→`fixUnderlayRect` (per-rect, 100+×/frame, during the IIFE). Intermittently aborted SudokuPad's init before `buildAllUI` (no buttons, board grey/black). `setTimeout(0)` / waiting for `load`+300 ms did **not** reliably fix it — the async re-apply was still racing init.
- ⚠️ **v3.16 (stable but wrong colours):** ripped the model out entirely; detect **and** group purely from the DOM (`getDomShadedRegionMap`): grey ≈full-cell `#underlay rect`s → flood-fill into contiguous blobs → `computeRegion4Colors`. Stable, but **flood-fill can't separate two distinct logical regions that touch** — in "We Live Here" all 4 touch transitively → all one colour. The feature looked broken.
- ✅ **v3.17 (fix):** keep v3.16's **DOM-only DETECTION** (the early/hot `puzzleHasShadedRegions` path stays Framework-free), but do the **GROUPING from the model, read SYNCHRONOUSLY** via `Framework.app.puzzle.currentPuzzle.cages` (`readModelExtraRegions`, guarded + try/catch). Filter: `unique && type∉{region,rowcol} && (style==='extraregion' || (hidden && sum==null && value==null))`; `cells` is a `"rXcY,…"` string (1-indexed → subtract 1). Require the model regions to **overlap** the detected grey cells, then `computeRegion4Colors` so touching regions differ. **No `getApp`, no promise, no `applySettings`-from-resolve.** Flood-fill stays only as a fallback for the (vanishingly rare) window where grey rects exist before `currentPuzzle` is set: the cache is non-`final` then, and a one-shot `setTimeout`→`fixAllUnderlays` upgrades it once the model lands. (SudokuPad parses `currentPuzzle` *before* rendering its underlay, so in practice the model is ready on the first paint that sees grey rects.)
- 🧰 **Decode a SudokuPad puzzle payload offline** with `pip install lzstring` (one-off): `/api/puzzle/<id>` returns `fpuz…` (f-puzzles) or `scl…` (SudokuPad/Sudoku-Maker), both `LZString.decompressFromBase64` of the JSON after a short prefix. The `scl` form uses unquoted keys + `t`/`f` booleans (not valid JSON); read it as text. This is how the hidden-unique-cage structure above was found without the browser.
- 🪲 **`SCRIPT_VERSION` (drives `window.spdrVersion`) had drifted to `3.11.0` while the `@version` header advanced.** `window.spdrVersion` was therefore useless for confirming which build is live. Keep `SCRIPT_VERSION` (≈line 209) in sync with the `@version` header on every bump. (Also: the Claude-in-Chrome tool redacts `window.spdrVersion` as "[BLOCKED: JWT token]" because `x.y.z` has two dots like a JWT — read it via char codes to bypass.)
- 🪲 **Don't trust rapid programmatic `location.reload()` in the `#variant=b` A/B harness for stability testing** — back-to-back reloads in the reused MCP window corrupt the harness/TM init and fail to load ~30-50% of the time *regardless of script changes* (confirmed: breaks even with the feature disabled). It looks exactly like a real regression. Test stability with **fresh tabs + single navigations** (a normally-loaded page is reliable), which matches the user's real setup.

## Action buttons via the paste path — `app.act` + undo groups (v3.18–3.19)

The Fill/Clear/Clear-All workers used to drive marks by `ensureMode()` + clicking the digit button — visible mode/button flicker. v3.18 swapped that for SudokuPad's own paste primitive: **`app.act({type, arg})`** applies a mark to the api-selected cells with no mode switch and no button click.

- 🔑 **`ToolToAction` (read off `Puzzle.ToolToAction`):** centre→`'candidates'`, corner→`'pencilmarks'`, normal→`'value'`, colour→`'colour'`. So `app.act({type:'candidates', arg:'5'})` toggles centre-5 on the selected cells. **It TOGGLES** (present→removed, absent→added) — so to *add* select cells missing the digit; to *remove* select cells that have it.
- 🔑 **Copy/paste is in-memory, NOT the system clipboard.** `FeatureCellPaste.handleCopy` stashes `this.copiedProps` (e.g. `{centre:['1'..'9']}`); `handlePaste` replays it via `app.act` wrapped in `groupstart`/`groupend`. Notepad can never see it. (The feature instance isn't reachable from `app`; don't bother hunting for it — just call `app.act` yourself.)
- 🔑 **`app.act({type:'groupstart'})` … `groupend` = one undo step.** Wrap a logical op in it so a single Ctrl+Z reverts the whole thing. Fill = 1 group, sweep = 1 group ⇒ the Fill button is 2 undos.
- 🪲 **THE TRAP (cost a whole v3.18→3.19 cycle): `groupstart` HIDES the `.conflict` CSS classes for as long as the group is open; they reappear on `groupend`.** The invalid-mark sweep finds its targets by reading `tspan.conflict`/`text.conflict` from the DOM — so if you open the group *before* collecting, you always collect **zero** targets and silently remove nothing ("removed 0 invalid marks" while conflicts plainly exist). **Fix: collect conflicts with NO group open, THEN open the group for the removals.** That's why the per-pass `_removeInvalidPencilmarksInternal` opens its own group *after* collection (not the multi-pass wrapper around it).
- ✅ Conflict classes are otherwise applied **synchronously** after `app.act` (no RAF/debounce wait needed) — and SudokuPad *does* flag a candidate against a **given** in the same house, not just against placed values.
- ⚠️ **Testing gotcha:** SudokuPad persists per-puzzle progress in `localStorage` (shared across the profile), so test marks survive reload and pollute the next run. Wipe with `app.select(app.puzzle.cells)` then `app.act({type:'clear', arg:'centre'|'corner'|'normal'|'colour'})` inside a group. Also, the action buttons (`#sp-fill-btn`/`#sp-clear-btn`/`#sp-clearall-btn`) sometimes need several seconds to build after load — poll for them.

## Beating DarkReader (the core problem)

DarkReader overrides SVG fills two ways:
1. **Generated CSS** — counter-rules in `<style class="darkreader darkreader--override">`, often the *same* specificity as ours but *later* in the document, so they win on cascade order.
2. **Inline attribute override** — adds `data-darkreader-inline-fill` plus an `!important` stylesheet rule driven by `--darkreader-inline-fill`.

- ✅ **The master key — inline style with `!important`:** `el.style.setProperty('fill', value, 'important')`. DarkReader does **not** fight `!important` inline styles. Used for given digits, Kropki dots, underlay fills, pencilmarks, cell colours.
- ✅ **CSS variables — restore via `html[data-darkreader-scheme="dark"]`** (specificity 0,1,1), which beats DarkReader's `:root` block (0,1,0) for `--cell-color-*`.
- ❌ **CSS `!important` alone does NOT work:** DarkReader generates an equal-specificity counter-rule that appears *after* ours and wins. Always use inline-style `!important` for fills, never a stylesheet rule.
- ❌ **Don't strip `data-darkreader-inline-fill` from all rects:** it breaks the textbg-rect fix. No blanket removal.
- ✅ **Turning a shading feature OFF must repaint DR's converted colour as a RESOLVED LITERAL** (v2.145 → corrected v2.149). Once we set an `!important` inline colour, DR gives up on the element **and clears its own `--darkreader-inline-*` var** (verified) — so `removeProperty('fill')` on disable leaves the raw, bright attribute colour, not DR's dim value. We must repaint DR's dark colour ourselves: `captureDrInline` stores it on `dataset.spdrDrFill`/`spdrDrStroke`, `restoreToDr` paints it back with `!important` on disable (it never touches the data-attribute, so the `data-darkreader-*` observer doesn't loop).
  - ❌ **The v2.149 bug fix corrects a v2.145 mistake: do NOT store/restore DR's var EXPRESSION** (`var(--darkreader-background-cfcfcf,#333739)`). Setting that as `style.fill` makes DR's observer see the `#333739` hex and re-convert it into a *text*-colour var (which for a dark input resolves *light*), nesting one level deeper and drifting **brighter** every toggle/redraw. That was the "disable makes it brighter / enabling region borders brightens the shaded areas" regression.
  - ✅ **Store a resolved literal rgb instead.** Resolve a colour expression via a throwaway `<span>` (`style.color = expr` → `getComputedStyle().color`); DR's vars live on `:root` so `var()` resolves synchronously through the cascade. Capture **once** (first clean value wins; the original colour never changes) so a later DR-derived value can't overwrite it. Deterministic fallback when the element's own DR var isn't present yet (scan-vs-DR race): DR publishes `--darkreader-<kind>-<hex>` root vars keyed by the element's stable original colour (`background` for fills, `text` for strokes), so build that reference from the `fill`/`stroke` attribute and resolve it. (a) restoring only the DR var *without* its data-attribute renders nothing — DR's rule is keyed on the attribute selector, which we removed. (b) our `!important` inline literal beats DR's stylesheet rule, so it wins whether or not DR's marker is still there.

## SVG rendering

- ✅ **`shape-rendering="crispEdges"` on our group** (fixed v2.82): the SVG is scaled ~1.333× (810px screen ÷ 608px viewBox), so strip boundaries land on fractional screen pixels; the browser anti-aliases the edge into the dark cell behind it → a thin dark fringe. crispEdges snaps boundaries to whole pixels and the fringe disappears.
- ⚠️ **crispEdges on mainGroup must NOT reach the cloned `path.cell-grid`** (fixed v2.146). The grid clone (z-order fix) is a child of mainGroup, so it inherited `shape-rendering:crispEdges`. crispEdges is right for the strip *rects* (kills the fringe) but wrong for the thin 1px grid lines: at ~1.33× scale each line is ~1.33px and crispEdges snaps it to 1px or 2px depending on sub-pixel position → some dividers render brighter/wider than others, with the rounding differing per browser (LibreWolf vs Brave hit different lines), and worst over shaded underlay cells. The original `path.cell-grid` was `shape-rendering:auto`. Fix: set `shape-rendering:geometricPrecision` on the *clone* to override the inherited crispEdges back to smooth. Lesson: anything thin/anti-aliased that we move into the crispEdges group needs its shape-rendering reset.
- ✅ **`stroke: none !important` on the 9 cage-box paths** (fixed v2.81) whenever any feature is active: those `#cell-grids path:not(.cell-grid)` paths have opaque-black stroke at z=8 and otherwise paint dark lines through coloured cells at 3×3 box boundaries. Restore the stroke when all features are off.
- ✅ **Cell-grid z-order fix** (v2.85): clone `path.cell-grid` as the *first* child of our group (so strips cover the clone) **and clear the original's `d` attribute** so it stops rendering at z=8. Restore the original from `dataset.spdrOrigD` on the next call.
- ❌ **Never use `display:none` (or any style change) on `path.cell-grid`:** the `#cell-grids` MutationObserver watches `attributeFilter: ['style']`. A style change fires it → infinite loop. Changing `d` does **not** trigger it — that's exactly why we clear `d` instead.
- ✅ **White-stroked `#cell-grids path:not(.cell-grid)` are author "grid-line erasers", not box outlines (v2.192).** Most puzzles' non-`.cell-grid` paths in `#cell-grids` are the dark `#000000` 3×3/region box outlines we suppress + redraw. But some authors lay a **`stroke="#FFFFFF"` width-3** path *over* a grid line to HIDE it — invisible white-on-white in light mode, used to carve irregular outer shapes / merge cells (e.g. the Gattai puzzle `y22px1zdae`: a 10×10 grid whose play area is two overlapping sub-grids, the rest erased). `fixCageBox` assumed black outlines, so it broke both ways: with region borders on it set `stroke:none` → **revealed** the grid line the author hid; with them off it restored DR's `--darkreader-text-000000` (#e8e6e3) → painted a **bright** thick line. **Fix:** detect a near-white stroke attr (r,g,b ≥ 240, a≠0) and, under DR, repaint it `var(--darkreader-background-ffffff, #181a1b)` at native width so it keeps **erasing** the (now light) grid line — exactly what SudokuPad's own dark mode does via `.setting-darkmode [stroke="#FFFFFF"]{stroke:var(--dm-black)}`. Discriminator is stroke COLOUR (white = eraser, dark = real outline), which is safe: a *visible* border is never white (invisible on the light default), so white in `#cell-grids` always means eraser.
- ⚠️ **`y22px1zdae`'s thick "region borders" are COSMETIC lines, not region/box data — deliberately left alone (v2.192).** The puzzle's visible region structure is drawn as 15 thick black cosmetic polylines in `#overlay` (`stroke=#000000`, `stroke-width=5.1`); there is **no** region/box structure in the model (the 20 `(none)` cages are just the row+col uniqueness constraints `r1c1-r1c10`/`r1c1-r10c1`, and the 4 dashed `#cages` paths are cosmetic overlap reminders). DarkReader already converts those black lines to light thick lines, so they render fine and read as borders against the 1px grid — they're just thicker, same colour. Our region-split feature can't "recognize" them because inferring regions from arbitrary cosmetic `#overlay` lines has no reliable signal (cosmetic black lines are decoration on countless puzzles) → high blast radius across the ~100 vetted puzzles. **Decision (confirmed with the user): leave them.** Anything we draw over them is more confusing than the already-fine DR rendering. So the region-split feature stays inert here by design; only the eraser fix above applies.
- ✅ **Region 4-colouring must use backtracking, not plain greedy (v3.12.0).** `computeRegion4Colors` builds the orthogonal-edge adjacency graph (planar → four-colour theorem guarantees a proper 4-colouring exists) and feeds BOTH features off one shared `regionColors[]` array (multi-colour borders use `palette[ci]`, Easy Shade fills use `fillPalette[ci]` — same indices, so they always match; the Easy Shade button only flips `regionColorFillEnabled`, it computes nothing itself). The original code did **plain greedy in row-major order**, which is *proper* (never colours two touching regions alike) but does **not** stay within 4 colours on irregular/jigsaw region layouts — greedy can spill to a 5th colour (index 4). The palette has only 4 entries, so `palette[4]` is `undefined` → the region's strips/fills render with **no fill = black borders** (the user hit this on real puzzles). Fix: greedy fast-path, and if it overflows past 0–3, run a backtracking proper-4-colouring (guaranteed to succeed for a planar graph; region counts are tiny so it's instant), plus a clamp-to-0 final guard so an index can never exceed the palette. Standard 3×3-box sudoku never tripped it (that graph is 2-colourable); only irregular regions did.
- ✅ **Disjoint/scattered sudoku regions need the puzzle MODEL, not geometry (v3.13.0).** Some puzzles (e.g. clover's "Scattered" `2xjf6aw4m2`) define one sudoku region whose 9 cells are **not contiguous** — scattered single cells that are one region for digit logic and must share one colour. `inferRegionsFromSVG` is pure geometry (flood-fill between border paths) and CAN'T know non-touching pieces are one region → it saw them as 9 separate regions in 9 colours. **The logical region list is in the runtime model:** `app.puzzle.currentPuzzle.cages`, filtered to `type === 'region'`, each with a comma-separated **1-indexed** RC-notation `cells` string (`"r2c2,r5c8,…"`). NB: `app.puzzle.cells[i]` does NOT carry a `region` field (only row/col/value/…); the resolved region membership lives only on the `currentPuzzle` cage list. Normal sudoku has **no** `type:'region'` cages (boxes are implicit) → map is null → geometry path unchanged. Implementation: `getModelRegionMap` caches `{ 'r,c': logicalId }` keyed by `location.pathname`; since the render path is sync but `Framework.getApp()` is async, first call returns null + kicks an async refresh that repaints once when the map lands (guarded by a `_pending` flag + key-match so no loop). `buildRegionGroups` collapses geometric pieces with the same logical id into one colour group; `computeRegion4Colors` now colours per GROUP and returns per geo-region (drawing code unchanged — disjoint pieces just get matching colours and each still draws its own border outline). ⚠️ **Forcing a DISCONNECTED region to one colour can break 4-colourability** (the four-colour theorem only guarantees 4 for *connected* regions) — so `computeRegion4Colors` falls back to ungrouped (geometry) colouring when no proper grouped 4-colouring exists, preferring a clash-free result over the disjoint match in that rare case. (Single-disjoint-region puzzles are usually still 4-colourable: the scattered region touches everything → just takes the 4th colour.) **v3.14.0: disjoint groups are PINNED to palette colour 3 (the 4th, orange)** so the common regions keep the earlier, more-frequent colours (red 0 > blue 1 > green 2) and the scattered region is the rare orange. `countComponents` flags a group as disjoint when its cells form >1 orthogonal component; `colourGraph` takes an optional `pinned[]` (fixed colour per node, rejects mutually-clashing pins, backtracks only the free nodes). Degrade order: pinned grouped → unpinned grouped → ungrouped geometry. (Pinning the scattered node to 3 forces all its neighbours off 3, so the rest must be 3-colourable with {0,1,2} — held for `2xjf6aw4m2`; if it ever doesn't, it falls to unpinned automatically.)
- ❌ **Dead end — `fixCellGrid`** (the v2.71–v2.80 clipping function): tried to clip cell-grid segments at region boundaries. Removed v2.83. The "white lines in black cells" it chased were actually the cage-box strokes (fixed v2.81) + sub-pixel anti-aliasing (fixed v2.82). v2.67 had neither `fixCellGrid` nor the white-lines problem — confirmed baseline.

## Region / shaded-region geometry

- ✅ **`path.cage-extraregion` only ever means a *checked* region (digit-uniqueness enforced), never cosmetic shading.** Verified via `window.convertedPuzzle.cages`: those paths come from cages with `style:"extraregion", unique:true, sum:45` (a real 1-9 no-repeat constraint). The cage `style` tally on a normal puzzle is `region` (the 9 boxes), `rowcol` (rows+cols), and `extraregion`. Purely decorative grey shading lives in a **separate `cosmetic` array** and renders through a different path — it does NOT get the `cage-extraregion` class. (`window.convertedPuzzle` is the handy decoded-puzzle global; the `/api/puzzle/<id>` response is `fpuz…` = LZString-compressed f-puzzles, not directly readable.)

- ⚠️ **Don't derive cell size from `cage-box` path coords.** SudokuPad draws each box outline with a *midpoint* vertex at the box centre — e.g. a 3×3 box of 64px cells emits `M0 0 L96 0 L192 0 …` (96 = box centre, not a cell boundary). - ✅ **Greedy 4-colouring, highest-degree-first** gives a valid adjacency-aware colouring for the grey-region touch graph (planar). Same approach as `computeRegion4Colors`; a 5th-colour fallback (reuse least-clashing) guards the rare non-planar case.

So anything it draws must not depend on state set earlier *in applySettings only* — e.g. the shaded-region clones recompute `assignExtraRegionColors(svg)` inside the `needShaded` block, because an observer-triggered draw would otherwise read stale/cleared `dataset.spdrExtraColorIdx` and produce **0 clones** (regions invisible). ## Kropki detection

The trap: SudokuPad uses the same `rect.textbg` class for Kropki circles **and** non-Kropki labelled circles (arrow constraints, operators).

- ✅ **Robust detection = THREE independent signals, all required (v2.184).** A Kropki-type edge dot is: (1) **shape** = a true circle; (2) **fill** = pure black `#000000` or pure white `#FFFFFF`; (3) **position** = centre on a cell border (`isOnCellBorder`). `isKropkiCircle` = signal 1 only; `isKropkiDotRect(rect,cs)` = 1∧2∧3 and is the single source of truth for "the colour fix owns this" (object shading + label-bg both skip `isKropkiDotRect`, so a dot is never double-claimed). The three signals were hardened together in v2.184 after three separate puzzles each exposed one gap:
  - ❌ **`rx ≈ w/2` alone does NOT prove "circle" — it also matches a PILL's short side (v2.184 fix for issue #2).** A vertical 2-cell double-arrow bulb is a `textbg` rounded rect w=48,h=112,rx=24; `rx ≈ w/2` (24≈24) passed, so it was mis-detected as Kropki, force-coloured white + given a `~`. (Horizontal pills w=112,h=48,rx=24 already failed since rx≠w/2 — that asymmetry is why only *some* bulbs broke.) Fix: a circle requires **`w ≈ h` AND `rx ≈ w/2`**. Regression puzzle `bill-murphy/20260526-david-and-goliath-sudoku`.
  - ❌ **Requiring the `feature-kropki`/`textbg` class missed CLASS-LESS dot circles (v2.184 fix for issue #3).** clover's "Diamond Ring" (`nipb00tmn1`) draws its white edge dots as bare `<rect>` circles in `#overlay` with **no class at all**. They failed `isKropkiCircle` (class gate) → never colour-fixed → DR left them dark; *worse*, they matched the white-fill arm of `LABEL_RECT_SEL` so `fixLabelRect` painted them the dark label-bg colour (the "black dot"), and the black stroke DR-inverted to white (the "white circle" ring), with no `~` injected (not detected as a bare dot). Fix: `isKropkiCircle` now accepts a geometric circle whose class is `textbg` **or absent** (rejects other *named* classes); the scans (`fixAllKropkiDots`/`rebuildKropkiLabels`/`HT.kropki`) widened to `… , #overlay rect, #underlay rect`, with the fill+position gates doing the real filtering. **Safe because the position gate (signal 3) is the discriminator** — this is the v2.164 philosophy, NOT a repeat of the reverted v2.166 "force-keep author white" (which had *no* positional signal). Verified 0 newly-grabbed / 0 dropped on both test puzzles before shipping. SudokuPad never emits `<circle>` (all rounded rects), so rect-only scanning is complete.
  - ❌ **Object shading and label-bg must SKIP Kropki dots, or a black dot reads as a "gray object" (v2.184 fix for issue #1).** `shouldShadeOverlayRect` only skipped grayscale **`textbg`** dots, so black **`feature-kropki`** dots (e.g. `264wvenhmu`, native Kropki in `#overlay`) fell through → counted as gray objects by `objFillSources`, so the **Gray object brightness** eyeball highlighted them (white dots were excluded only incidentally, by the near-white rule — hence the black/white asymmetry the user noticed). The render was correct only because `fixAllKropkiDots` runs *after* `fixAllUnderlays` and overwrites. Fix: `shouldShadeOverlayRect`, the `#underlay` arms of `fixAllUnderlays`/`objFillSources`, the observer's underlay branch, and `fixLabelRect`/`HT.labelBg` all bail on `isKropkiDotRect` — so apply + highlight agree (foolproof principle) and the Kropki section is the dots' sole controller (Kropki off ⇒ they fall to DR, not to object shading).
- ✅ **Disambiguation rule = POSITION only (`isOnCellBorder`), v2.164.** A circle (black *or* white) is a Kropki-type dot **iff its centre sits on a cell border** — a gridline in one axis, mid-cell in the other. That's the defining geometry of any edge clue (Kropki / X-V / operator dots like Makodoku `+`/`×`). The colour fix (`fixKropkiDot`), label injection (`rebuildKropkiLabels`), **and** the `HT.kropki` highlight all gate on it (foolproof principle). Off-border circles are left to DarkReader: **quadruples** sit on a grid **corner** (gridline in *both* axes), arrow bulbs / cosmetic circles at a cell **centre**, line endpoints wherever a path ends.
  - ❌ **The old v2.137 OR-clause `svgHasBlackKropkiCircle` was the bug (removed v2.164).** It said "if the puzzle has *any* black Kropki dot, treat *all* white circles as Kropki" — so on a puzzle with both a black dot and a quadruple (test puzzle `pbwqsppuho`), the quadruple's white corner-circle got force-coloured **and** a `~` label drawn over its existing 1-4 digits. Position alone suffices: Makodoku's all-white border dots still pass `isOnCellBorder` (which is why the black-dot anchor became redundant once the border test was added in v2.137). Note `getKropkiAdjacentText` did *not* catch the quadruple's digits (they're not an immediate text sibling), so the no-text gate alone didn't save it — the position gate does.
- ✅ **Two-tier shape test (each then gated by `isOnCellBorder`):** `isKropkiCircle` (shape) → colour fix; `isKropkiRect` (shape *and* no text sibling) → label injection. Labelled Kropki circles (Difference / Ratio / Makodoku operators) get their colour fixed but are **not** re-labelled, and the no-text-sibling gate also means the rotate-on-horizontal-border setting never touches them.
  - ❌ **`getKropkiAdjacentText` must skip the `textbg` halo between a dot and its value, or labelled dots get a spurious `~` (v3.30.0, clover "Same Difference" `0jyxu79n6q`).** That puzzle's DOM per clue is `rect(white dot) → rect.textbg(fill:none, w/h=-2) → text("2")`. The lookup checked only the dot's *immediate* next sibling, hit the `textbg` rect, concluded "bare dot," and injected a rotated `~` over all 28 values. Fix: the sibling walk now skips `rect.textbg` (and our own injected labels) to reach the real value text. Safe for genuinely bare dots (after the textbg comes the next clue's *circle* → `null` → still labelled) and for quadruples (off-border, excluded by the position gate regardless). **Surfaced by `spdrGapScan`** — the gap badge flagged the invisible `textbg` strokes, which traced straight to this bug; the scan also gained a geometry guard so non-rendering rects (w≤0 or h≤0) no longer false-flag.
  - ❌ **The value text often isn't a DOM sibling AT ALL — match it by POSITION (v3.31.0).** SudokuPad batches edge-clue values in a separate pass within the overlay group, so for **Ratio** (fractions `1⁄3`, `5⁄7` — `3y38nrs34s`) and **XV / Roman sums** (`XIII`/`VIII` — `ed0mko9d0b`, `ogcall10hl`, `rMD9P4HFjP`, `t12fc7v8bl`, `rjl0oqocet`) the dot and its value share no sibling relationship — the only stable link is that the value's anchor sits on the dot's **centre** (`text-anchor:middle`, a few px down for the baseline). The sibling-only lookup missed them → every labelled dot got a spurious `~` *and* its value colour was never pinned, so off-white numerals vanished on the white disc ("inverted" look). Fix: `getKropkiAdjacentText` falls back to `getCenteredKropkiText(rect)` — scans `#overlay/#underlay` non-spdr text for one anchored within `dx≤10, dy≤16` of the dot centre (well under one cell, so a neighbour ≥1 cell away never matches; scoped to overlay/underlay so grid digits/pencilmarks are never grabbed). This one helper fixes both the spurious label (via `isKropkiRect`) and the unpinned colour (via `fixKropkiDot`'s `adjText`). Bare dots stay bare (no centred text → `null` → still get `~`).
  - ✅ **Labeled dots get a FIXED look fully decoupled from the Kropki controls (v3.32.0 → v3.33.0).** A dot carrying a printed value (Ratio fraction, Difference number, XV/Roman sum, operator symbol) uses fill only cosmetically — the value carries the meaning. `fixKropkiDot` handles them in a dedicated `if (adjText) { … return; }` block placed *before* the `kropkiFixEnabled` gate: **black disc, NO outline ring, white text**, set unconditionally. None of the Kropki controls touch them — not the outline toggles, not the label text/size/rotate (they never get an injected label — `isKropkiRect` is false), not even the master enable. Native dark renders these broken (off-white text on a white disc), so it's a correctness fix, not a style toggle. **BARE dots** fall through to the old control-driven branch and keep their semantic fill (white = consecutive, black = 2:1), so normal Kropki puzzles are untouched. The `adjText` signal (sibling + positional, see above) is the sole gate. (v3.32.0 first did this *with* a white outline; v3.33.0 dropped the ring per request.)
  - ✅ **Off-border Kropki-type clue SHAPES get their own controller, `isKropkiClueShape`/`fixKropkiClueShape` (v3.34.0).** Pure black/white circles at a cell CORNER/CENTRE (quad / clock-face dots) and DIAMONDS (rotate-45 squares, Kropki-cage ratio markers) bypass `fixKropkiDot` (off-border / not a circle), so they fell to object-shading (black→grey `#999`) and label-bg (white→dark) and came out inverted. New dedicated pass renders them in the kropki convention: **solid** shape → author colour + contrasting outline (black→black+white outline, white→white+dark outline); shape **with a centred glyph** (clock dial) → ring (dark fill + light outline + light glyph) so the arrow shows and both dial colours read as matching light rings. Object-shading (`shouldShadeOverlayRect`, `objFillSources`) and label-bg (`fixLabelRect`, `HT.labelBg`) all skip `isKropkiClueShape` (sole-controller / foolproof pattern). **Size gate `w ≤ cs*0.5` is load-bearing**: it includes kropki-sized dots/diamonds (~16–25px) but EXCLUDES digit **quadruples** (~38px white corner-circles holding 1–4 digits — `pbwqsppuho`), which must keep their native look. Fixes `rl63gh666j` (clock face), `fah0d6vyk0` (diamonds), `6ltsd1fax2` (consec quads).
  - ℹ️ **"Pills treated as black Kropki" (`6thevyefk2` Clock Sudoku) was already resolved by the v2.184 `w≈h` rule** — the clock pills are `240×48` `textbg` rects (w≠h → rejected), the `:` separators are the author's own text, and nothing is force-coloured as a dot. Verified clean in v3.30+.
- ❌ **Naively forcing _all_ white circles white does NOT work:** it inverts arrow-constraint puzzles (white circles carrying arrow glyphs like ↖, sitting at cell centres) into black-outlined shapes, overriding DarkReader's correct dark-mode rendering. (The bug behind the v2.119 fix; regression puzzle `jhrb0vsbnk` — verified still off-border/untouched under the v2.137 edge rule.)

## Lines (thermos, palindromes, renban, whispers, arrows…)

- Every line-type clue renders as a stroked `<path>` in `#arrows` (`fill=none`). A thermometer is **two separate elements**: the **bulb** is a rounded `rect` in `#underlay` (caught by Object shading as a *fill*), and the **shaft** is one of these `#arrows` paths. They share a source colour (e.g. `#CFCFCF`). DR leaves the line near-white in dark mode → a mismatch against shaded fills.
- ⚠️ **X-diagonals (`sudokuX`) live in `#overlay`, not `#arrows`** — a stroked `<path>` with its own colour (e.g. `#34BBE6`). `fixAllLines` is scoped to `#arrows`, so it does **not** shade diagonals: they keep their native colour and are left to DR. Fine for a saturated diagonal (reads clearly on dark); a near-grey diagonal could in theory stay too bright under object-shading, but not observed. Confirmed on test puzzle 2 `3x3zm2co6o`. Targeting just diagonals is fragile (no distinguishing class), so deliberately left alone.
- ⚠️ **Line ENDPOINT circles** (between/lockout lines) are `rect`s in `#overlay`, not `#arrows`. Authors give them different alphas on purpose: solid bulbs `#FFFFFF`, see-through bulbs `#ffffff80` (so the line reads *through* the disc). Both match `LABEL_RECT_SEL`'s case-insensitive white-prefix arm (v2.196) and are darkened by the label-bg fix; coloured ones (e.g. diamond `#e7e6ff80`) go through object-shading instead. **`fixLabelRect` preserves each rect's authored alpha** (v3.8.0: `applied = labelBgOpacity × srcAlpha`) so `#ffffff80` stays at 0.5 and the line shows through — matching DR (DR darkens a fill while keeping its alpha). Without that, every white-prefixed rect flattened to one opaque colour and translucent bulbs wrongly blocked what was under them. Confirmed on `pbwqsppuho` (double-arrow bulbs solid, between-line bulbs see-through). **DR's natural per-bulb difference is just the author's alpha, NOT a z-level thing** — all four bulbs are in the same `#overlay` layer.
- ✅ **Shade the line stroke** with the same Object-shading transform as fills (`computeObjectShade`, applied to the stroke in `applyLineStroke`). Gray lines follow the linked Gray slider, colored lines the Brightness/Opacity sliders. Stroke **width is never touched** — only stroke colour + opacity.
- ⚠️ **Scope is now ALL stroked `#arrows` paths** (v2.140, `fixAllLines`/`isLineStroke`). The old v2.122 rule matched only shafts whose stroke equalled a `#underlay` bulb fill colour (`getBulbFillColors`/`isThermoShaft`, removed) — so **bulbless** line puzzles (palindromes/quads like `9p6eahqmux`, which have *zero* `#underlay` rects) fell through to DR and stayed bright. ## Gray vs colored object shading (v2.140)

- ✅ **Gray and colored need separate sliders.** `shadingTransform` forces **full saturation** for colored objects (vivid at L=0.4) but returns plain `rgb(L·255 …)` for gray (saturation 0) — genuinely dark at the same number. So one slider can't serve both. Each side has its own brightness + opacity keys (color `underlayLightness`/`underlayOpacity` @ 0.4, gray `underlayGrayBrightness`/`underlayGrayOpacity` @ 0.6). `computeObjectShade(c)` routes on `isGrayColor` (saturation `< 0.08`) and returns `null` when the relevant control is disabled (caller then clears its inline style → back to DR). Used by both fills (`applyShadingFill`) and lines (`applyLineStroke`).
- ✅ **Combined vs separate is UI-only (v2.141).** The **"Control opacity and brightness separately"** checkbox (`underlaySeparateBrightnessOpacity`, default off) just swaps which rows show: off → one *combined* slider per side that writes both axis keys to the same value (`makeRangeRow` `extraKeys`/`extraEnabledKeys`); on → the two axes as separate sliders. The shading code never reads the flag — it always reads the 4 keys, which combined mode merely keeps equal. Entering combined mode forces `opacity := brightness` (both sides) so the single slider matches what's applied.
- ⚠️ **Several rows share a setting key** (combined-color and separate-color-brightness both bind `underlayLightness`), so `makeRangeRow`'s per-row `controlSyncers[key]` registration clobbers. Fix: the object-shading `subBuilder` overrides `controlSyncers` for all its keys with one `refreshAll` that re-syncs **every** row (via each row's `spdrRefresh`) and re-applies the visible-row mode — so both the section ↺ and the bottom "Reset all" restore the UI correctly.
- Shape **outlines** (`applyShapeStroke`) follow the same combined/separate toggle as colored/gray (Border brightness + Border opacity, `underlayStrokeLightness`/`underlayStrokeOpacity` @ 0.75). Brightness preserves the element's hue via `shadingTransform`; opacity became configurable in v2.143 (was hardcoded `stroke-opacity: 1`).
- The section ↺ deliberately **omits** `underlaySeparateBrightnessOpacity` from `resetKeys`, so resetting a section's values doesn't flip the combined/separate checkbox; only the bottom "Reset all" (which copies all of DEFAULTS) restores it.
- ✅ **Coloured cosmetic shapes keep their colour; only white/grey label boxes darken (v2.165).** `rect.textbg` is shared by Kropki dots, white label boxes (cage sums / little-killer / XV operators) AND author-**coloured** cosmetic shapes (e.g. a light-blue `#80E5FF` circle or `#5EE7FF` box behind a digit). `fixLabelRect` used to flatten *all* matches to the dark label colour, so coloured cosmetic shapes rendered **black**. Fix: `fixLabelRect` skips SATURATED fills (`!isGrayColor`), and `shouldShadeOverlayRect` now **shades** coloured `textbg` (still skips white/near-white + grayscale textbg = Kropki/grey labels). So coloured cosmetic shapes route through Object shading (dimmed, hue preserved) like any other overlay shape; white/grey label boxes still darken. `objFillSources` reuses `shouldShadeOverlayRect` so the `objColored` highlight follows automatically; `HT.labelBg` filters out the coloured ones to match `fixLabelRect`. Found on test puzzle 2 `3x3zm2co6o`.
- ✅ **`fill="none"` (or transparent) label rects are INVISIBLE anchors — never darken them (v2.176).** Authors reuse a `rect.textbg` purely to position a label (e.g. a little-killer "12" outside the grid sitting *over* an arrow). Its author fill is `none`, so it renders nothing and DR ignores it (no `data-darkreader-inline-fill`). `fixLabelRect` was painting it our `#181A1B` label-bg `!important` → an opaque dark box that **covered the arrow beneath it**. The tell that it's our doing, not DR's: turning the Label-background checkbox off (`removeProperty('fill')` → reverts to `fill="none"`) made the box transparent and the arrow reappear; and the rect carried **no** DR marker/var. This is also why these differ from the white cage/little-killer label boxes (`fill="#FFFFFF"`), which DO need darkening — same `textbg` class, different fill attribute (`none` vs white). **Label backgrounds are darkened by TWO mechanisms — both must skip `none`:** (a) inline `fixLabelRect` (skip when raw fill attr is `none` or parses `a===0`; `parseColor("none")` returns `null` like an absent attr, so test the **raw attribute string** `=== 'none'`), AND (b) an injected **CSS rule** in `buildCSS` (`rect.textbg { fill: <bg> !important }`). Fixing only (a) made the box go **light grey `#D8D5CF`** instead of dark, not transparent: DarkReader **rewrites our injected CSS rule** (sees the dark `#181A1B` fill in a stylesheet → inverts it to light), so with the inline gone the DR-lightened CSS rule took over. (The inline `!important` had been overriding DR's counter-rule — the "inline beats DR, stylesheet loses to DR" lesson in action.) So the CSS selector also needs `rect.textbg:not([fill="none"])` (both the dark-scheme and purple-theme copies). With neither mechanism touching it, the rect falls back to its `fill="none"` attribute → transparent → arrow shows. (8 such rects on test puzzle `pbwqsppuho`.)
- ❌ **Don't force-keep author white on bare cosmetic shapes (tried v2.166, reverted v2.167).** A throwaway test puzzle had a white cosmetic circle (class-less white rect) that DR darkened to near-black; we briefly added `isLineEndpointCircle` (geometry: is the circle within ~1 radius of an `#arrows` line end?) to tell standalone cosmetic circles from between-line endpoints, then force-kept `#ffffff` on the cosmetic ones. **Reverted: narrow in intent but BROAD in reach** — it force-whitened *every* class-less white shape on *every* puzzle, risking regressions across the ~100 already-vetted puzzles (a previously-fine darkened white shape would suddenly glare white). Deeper point: **there is no broad signal** that distinguishes "author wants this white" from "fine to darken" — DR's broad answer is *darken*, so preserving white inherently needs a per-anomaly heuristic or an explicit user toggle, neither worth it for a one-off. **Conclusion:** bare white cosmetic shapes follow the broad rule and **darken** like any white non-underlay rect. (The v2.165 COLOURED-shape fix stays — it's broad and correct: coloured cosmetics route through Object shading with hue preserved.)

## Settings-clarity highlight overlay (v2.154–v2.163)

The 👁 hover-icons that show "what does this control affect" on the board. Architecture + the gotchas that took several iterations:

- ✅ **FOOLPROOF PRINCIPLE — a control's highlight must enumerate the SAME elements via the SAME predicates its apply code touches**, never a hand-written parallel selector (those drift and were wrong repeatedly). Object-shading getters reuse `objFillSources`/`objLineSources` (= `fixAllUnderlays`+`fixAllCagePaths`+`fixAllLines`) and route via `computeObjectShade` rules; `labelBg`=`LABEL_RECT_SEL`, `kropki`=`isKropkiCircle` filter. Consequence to remember: **lines route under the colored/gray sliders (by stroke colour), and "Border" = shape OUTLINES only** — not lines. And labelBg legitimately overlaps kropki on the circular `textbg` dots (label-bg recolours those too) — that overlap is correct, not a bug.
- ✅ **Trace real geometry, never a bounding box.** A single `<path>` (e.g. the whole grid) has a bbox = the whole board, so a bbox outline looked like just the outer square. Clone the actual element into an overlay `<svg>` and stroke it.
- ✅ **Use `el.getScreenCTM()`, NOT `getCTM()`.** getCTM maps to the board's viewBox space → fragile to match. getScreenCTM maps element-local → CSS px; put clones in a full-window overlay svg (`viewBox 0 0 innerW innerH`, `preserveAspectRatio none`) and they land exactly over the original regardless of viewBox/scale/scroll. Map raw points with the same matrix `{x:a·ux+c·uy+e, y:b·ux+d·uy+f}`.
- ✅ **De-duping adjacent shaded cells = occupied-cell-SET perimeter, NOT edge-counting.** Shaded cells are small INSET rects, so outlining each drew parallel lines between neighbours. Edge-count-and-drop-if-2 *seemed* right but broke when a cell holds multiple shaded elements (underlay tint + overlay rect): a perimeter edge counted twice got dropped, interior lines survived. Correct approach: build the set of occupied grid cells (snap each axis-aligned rect to its cell via `getGridCellSize`), then draw an edge only where the neighbour cell is absent. Robust to N elements/cell. Rotated rects (diamonds) + non-rects trace individually.
- ✅ **Paint modes** (`styleNode`): default **stroke** (outline) is what the user wanted for objects/selection — a translucent **fill** overlay read as "solid shading over the object" and was rejected *except* for thin shapes: `PAINT.regMulti='fill'` (multi-color border strips are 4px rects → fill = one clean bar, outline = doubled lines). `PAINT.cellColors='bbox'` outlines the cell square so a 2-colour cell's diagonal split isn't traced (the diagonal = top half of the colour boundary).
- ✅ **Click = pulse the highlight, not toggle the effect.** The wanted click behaviour is a gradual brighten→dim of the *highlight outline* (`spdrHi.pulse`, `@keyframes spdrHiPulse` ending at full opacity so it coexists with the steady hover-hold). An earlier on/off toggle of the real setting (`blinkEffect`) was wrong — removed.
- ✅ **Icons stay usable in a disabled section** via `pointer-events:auto` on the icon (a child re-enables pointer events under the section's `pointer-events:none` dim) — lets the user preview before enabling. Caveat: sections using `collapseWhenDisabled` (Region borders) hide their sub-rows entirely, so those icons vanish when off — not yet addressed.
- ✅ **Hover "examples" are safe because they're overlay-only or transient UI.** Pencilmark examples (`drawPencilExample`) draw demo digits into the overlay svg (`spdrHi.addText`) — they NEVER touch the puzzle model, so zero risk. The selection example is the one real-state action: `app.select(cells)` (selection is transient UI, intentionally left selected). Writing real pencilmarks via the app was avoided (would pollute solve/undo).
- **DOM facts (this codebase):** board svg = `#svgrenderer` (it *is* the `<svg>`); tool buttons `#control-colour`/`#control-centre`/`#control-corner`; cells are 64px full-cell `#underlay` rects but object-shading decorative squares are inset `#overlay` rects; multi-color region borders = many thin strip rects marked `data-spdr-kind="multi"` (center=path, cell=grid clone). Native `#cell-colors` opacity = CSS `var(--cell-color-opacity)` = **0.6** (so `cellColorsOpacity` default is 0.6 — enabling at default is a no-op).

## Centering an injected glyph on a Kropki dot (v2.183 — alphabetic baseline + geometricPrecision; supersedes v2.175 raster + v2.181 middle-baseline)

Centering the `:`/`~`/custom label *visibly* inside its circle is harder than it looks — `text-anchor:middle` + a baseline only center the glyph's **line box**, not its visible **ink**, and the gap differs per glyph. **There is no reliable RUNTIME way to measure ink here** — every measurement path has a fatal flaw, so do not try again:

- ❌ **`getBBox` / `getBoundingClientRect`** return the line/cell box (~19px tall for a `~` at font-size 16), not the ink. They report the label "perfectly centered" while it visibly isn't.
- ❌ **Canvas `measureText` `actualBoundingBox*`** metrics don't match SudokuPad's SVG rasterizer — the SVG renders the same glyph ~2px taller/offset. (That's the *metrics* API. Rasterizing with `ctx.fillText` + a `getImageData` pixel-scan — used for one-time calibration, below — IS reliable; it renders the real font outline. The two are not the same thing.)
- ❌ **`<img>`-isolated SVG** (clone the label into a standalone `<svg>`, rasterize via `new Image()` → canvas) **falls back to a different font** than the page (notably for `~`), so the measured ink lands in the wrong place.
- ❌ **The v2.175 "measure the actual render" approach** built on that isolated-SVG raster, cached per font signature and re-applied async. **Removed v2.181 — two fatal problems:** (a) `rebuildKropkiLabels` re-runs on *every* SudokuPad node insertion, so the async result raced the rebuild; (b) when measurement went wrong it didn't fail small — in v2.180 it **flung the colon to dx≈−12, dy≈−11 on a ~21px dot**, i.e. clean out of its circle (the exact "symbols aren't even in the circle" bug). Pixel-perfect *automatic* centering of arbitrary glyphs is not achievable here.
- ✅ **What works: a FIXED per-glyph nudge off the ALPHABETIC baseline, applied synchronously** (`centerKropkiLabel` + `KROPKI_INK_NUDGE`). Force `dominant-baseline:alphabetic` inline-`!important` on the label (beats SudokuPad's `middle` CSS rule, which a presentation attribute can't), then set the anchor to the circle center plus `{dx,dy}` in **fractions of font-size**. No measurement, no async, no cache → rebuild-safe by construction.
  - **Why the alphabetic baseline — the v2.182 CROSS-BROWSER fix.** v2.181 centered off the `middle` baseline; it looked right in **Blink (Chrome/Brave)** but **Gecko (Firefox/LibreWolf) shifted the ROTATED glyphs left.** `dominant-baseline:middle`/`central` require a font-metric computation (x-height / em-center) that the two engines do slightly differently — an unnoticed *vertical* difference upright, but the `rotate(90,cx,cy)` on horizontal-border dots maps the vertical axis onto the horizontal one, turning it into a visible *horizontal* shift (Gecko only). The **alphabetic baseline needs no metric computation — every engine agrees on it** — and a glyph's ink height above it is a pure font-outline property (same font everywhere), so a fraction calibrated in one browser centers it in all of them, both orientations. (`text-anchor:middle`, used for horizontal centering, is *also* engine-consistent — which is why only the baseline/rotated axis ever broke.)
  - **Why geometricPrecision — the v2.183 PER-DOT + cross-engine fix.** At the small label size (~16px) the font is **hinted**: each glyph's outline is grid-fitted toward whole device pixels, bending its ink ~0.5px sideways / ~0.7px down off true geometry — by an amount that depends on the dot's **sub-pixel position** (so every dot drifts differently; ~0.8px vertical spread measured across phases) AND on the **engine's hinter** (so Blink ≠ Gecko). The vector Kropki circle behind it is NOT hinted, so the glyph slides off-center, inconsistently — the "even within one browser they aren't consistent" symptom. Forcing **`text-rendering:geometricPrecision` inline-`!important`** makes the live SVG render the true unhinted outline — identical on every dot and every engine. No other lever disables hinting: `optimizeSpeed`, rendering a large glyph and scaling it down, etc. were all tested and do **not** help.
  - ⚠️ **geometricPrecision is honored by the LIVE inline-SVG paint, NOT by an `<img>`-decoded SVG.** A serialized `<svg>` → `new Image()` → canvas test shows `auto == geometricPrecision` and is **misleading** — that rasterization path ignores `text-rendering`. The real board (inline SVG painted in the document) *does* apply it. Verify the effect on the live board / with the user, never via an img-raster proxy. (This wrong proxy cost a cycle in v2.183 — I nearly concluded geometricPrecision was a no-op.)
  - **Calibrated values = ink-center height above the baseline, measured at LARGE size** (≥400px, where hinting vanishes → the same unhinted geometry geometricPrecision renders): `:` → `dy 0.275`, `~` → `dy 0.324`, unknown glyphs `dy 0.35`; all `dx 0` (the hinted ~0.5px left-shift is a hinting artifact geometricPrecision removes). `cy + dy·fs` drops the baseline below center so the ink lands on center. **Tweak a value if a glyph looks off — that's the intended knob.**
- ⚠️ **Rotation works in every engine *because* centering no longer depends on an engine-specific baseline.** Apply the nudge in the **unrotated** frame (set `x/y`; `rotate(90,cx,cy)` pivots around the circle center). Ink centered on `(cx,cy)` stays centered under rotation about `(cx,cy)`, so one nudge serves both orientations. The v2.181 rotation bug was never the rotation math — it was the baseline divergence the rotation merely *exposed*.
- 🔍 **Calibrating without screenshots (the reliable way).** Measure the ink via **canvas**: `ctx.font='600 <fs>px Tahoma, …'; ctx.textBaseline='alphabetic'; ctx.textAlign='center'; ctx.fillText(glyph, ax, ay)`, then scan `getImageData` for the ink bbox. Canvas `textBaseline:'alphabetic'` **is** the SVG alphabetic baseline, and it rasterizes the real page font, so `(ay − inkCenterY)/fs` is exactly the `dy` to use — and it's browser-independent (pure font outline), so one measurement serves all engines. Big `fs` (e.g. 400) for precision. (Bonus: the CDP/`Page.captureScreenshot` pipeline wedged mid-session — canvas needs no screenshots at all. Don't resurrect the isolated-`<svg>`→`<img>` raster; it had the font-fallback bug. Visual cross-check, when screenshots work: draw an isolated `position:fixed` overlay with the glyph big at `dominant-baseline:alphabetic` + a pixel ruler; the `zoom` tool crops 1:1 so the glyph must be physically big.)
